# DAG 工作流引擎 — 项目开发日志

> 记录时间：2026-07-14
> 项目定位：一个 Plan-and-Execute 架构的 DAG 工作流系统，支持可视化规划、流式执行、AI 辅助修改

---

## 一、项目背景

### 1.1 为什么做这个？

在日常的 AI Agent 使用中，发现了几个痛点：

1. **执行过程不透明**：Agent 在跑的时候，你不知道它在做什么、卡在哪一步、有没有死循环
2. **无法干预规划**：Agent 自己规划的路径可能不是最优的，你想手动调整但没有好用的工具
3. **缺少流程复用**：同一个任务每次都要从头跑一遍，无法保存和复用之前规划好的工作流
4. **观测能力缺失**：Token 花在哪里了？哪个步骤最耗时？工具调用了多少次？这些都看不到

于是决定做一个 DAG 工作流系统，把「规划 - 执行 - 观测」三个环节打通。

### 1.2 技术选型

| 层级 | 技术 | 原因 |
|------|------|------|
| 前端 | React + TypeScript + ReactFlow | ReactFlow 提供了优秀的 DAG 图可视化能力 |
| 后端 | Python + FastAPI | 生态好，与 LLM 相关的库丰富 |
| 执行引擎 | 自研拓扑排序 + 并发调度 | 不想被 LangGraph 等框架绑定 |
| 通信 | SSE (Server-Sent Events) | 流式输出体验好，实现简单 |

---

## 二、遇到的问题与解决方案

### 问题 1：网页检索工具太慢，体验差

#### 问题描述

最开始使用的网页检索工具（`web_search`）耗时非常长，经常一个搜索请求要等 10-30 秒才能返回结果。

这导致了几个问题：
- 用户以为系统卡住了，实际上还在跑
- 整个工作流的执行时间被拉长，耐心消耗殆尽
- 对比测试时，一个搜索节点的耗时占了整个任务的 80%

#### 思考过程

1. **是不是我们调用方式有问题？** → 检查了 API 调用参数，没有发现明显问题
2. **是不是网络原因？** → 排除，其他工具调用正常
3. **是不是应该换一个检索工具？** → 是的，这个工具本身的响应速度就是硬伤

#### 解决方案

- **更换检索工具**：换成了响应速度更快的搜索 API
- **增加超时控制**：每个搜索请求设置合理的超时时间，避免无限等待
- **结果缓存**：相同的搜索 query 在短时间内复用缓存结果

#### 效果

- 搜索响应时间从 10-30 秒降低到 2-5 秒
- 用户体验明显改善，不再以为系统卡住了

### 问题 2：没有流式输出，不知道系统有没有在运行

#### 问题描述

最开始的实现是「请求 - 等待 - 返回结果」的传统模式。用户点击「执行」按钮后，页面就进入了「加载中」的状态，没有任何中间反馈。

这导致了几个问题：
- **不确定性焦虑**：用户不知道系统是在正常运行、还是卡住了、还是崩溃了
- **进度不透明**：一个有 8 个节点的工作流，跑到第几个了？完全不知道
- **长时间等待的烦躁**：等了 30 秒还没结果，用户可能就把页面关了

#### 思考过程

1. **要不要做个进度条？** → 简单的百分比进度条不够，因为每个步骤的内容用户也想知道
2. **要不要做日志输出？** → 对，实时日志 + 节点状态更新，这个组合最清晰
3. **用什么技术实现？** → WebSocket 功能最全但实现复杂，SSE 对于单向流式输出足够了

#### 解决方案

- **引入 SSE (Server-Sent Events)**：后端通过流式接口逐事件推送
- **定义事件类型**：区分 `node_start`、`node_end`、`tool_call`、`llm_call`、`thinking` 等不同事件
- **前端实时更新**：
  - 节点状态：pending → running → completed / failed
  - 日志面板：实时追加每一步的输出
  - 节点详情：点击节点可以看到该节点的完整输出

关键实现片段：

```python
# 后端：通过 SSE 流式推送事件
async def event_stream():
    for event in executor.execute_streaming(graph):
        yield f"data: {json.dumps(event, ensure_ascii=False)}\n\n"
```

```typescript
// 前端：逐事件处理并更新 UI
const reader = response.body?.getReader()
while (true) {
    const { done, value } = await reader.read()
    if (done) break
    
    const event = JSON.parse(line.slice(6))  // 解析 "data: {...}"
    
    if (event.type === 'node_start') {
        setNodeStatuses(prev => ({ ...prev, [event.node_id]: 'running' }))
    } else if (event.type === 'node_end') {
        setNodeStatuses(prev => ({ ...prev, [event.node_id]: event.status }))
    }
    // ... 更多事件处理
}
```

#### 效果

- 从「黑盒等待」变成了「透明可视」
- 用户可以实时看到：哪个节点在跑、用了什么工具、输出了什么内容
- 即使执行时间长，用户也知道系统在正常工作

### 问题 3：编辑工具时不知道有哪些工具可用

#### 问题描述

用户点击「编辑工具」后，看到的只是一个空白的文本输入框。系统没有告诉用户：
- 有哪些工具是可用的？
- 每个工具叫什么名字？
- 每个工具是做什么的？

结果就是用户只能瞎猜，或者跑去看代码找工具列表，体验非常差。

#### 思考过程

1. **要不要做个下拉选择？** → 工具不多的时候可以，但以后工具多了体验不好
2. **要不要做个标签/按钮列表？** → 对，每个工具一个按钮，点击添加/移除，还能显示 tooltip 说明
3. **还要保留文本输入吗？** → 保留，因为有时候用户想手动输入（比如知道确切的工具名）

#### 解决方案

- **工具选择器 UI**：在文本输入框下方，显示所有可用工具的按钮列表
- **每个按钮**：显示工具名称，悬停显示工具说明
- **交互**：点击按钮切换选中/未选中状态，同时同步更新文本框内容
- **视觉反馈**：选中的工具高亮显示（蓝色边框 + 淡蓝背景）

#### 效果

- 用户不用记工具名了，看着按钮点就行
- 新用户也能快速理解每个工具是做什么的

---

### 问题 4：只有删除节点，没有新增节点

#### 问题描述

DAG 图的节点操作只有「删除」，没有「新增」。用户想加一个节点只能重新规划整个工作流，非常不灵活。

#### 思考过程

1. **新增节点需要哪些信息？** → 节点类型（RESEARCH/REVIEW/EXECUTE）、节点标题
2. **节点 ID 怎么生成？** → 自动生成，取现有最大 ID + 1
3. **边（依赖关系）呢？** → MVP 阶段先不强制连边，让用户自己后续拖拽连接

#### 解决方案

- **新增「+ 新增节点」按钮**：放在工具栏
- **新增节点模态框**：选择节点类型 + 输入节点标题
- **自动生成节点 ID**：如现有最大 ID 是 node_3，新节点就是 node_4
- **节点初始状态**：默认 EXECUTE 类型，用户可以切换

#### 效果

- 用户可以灵活地往工作流里加节点了
- 不用因为想加一个节点就重新规划整个流程

### 问题 5：无法从指定节点重新执行

#### 问题描述

最开始的「继续执行」功能，逻辑是「跳过所有已完成的节点，只执行未完成的节点」。

但用户的实际场景是：
- 工作流已经全部执行完成了
- 发现中间某个节点（比如 node_2）的输出有问题
- 想修改 node_2 的指令，然后从 node_2 开始重新执行（node_2 及下游都重跑）

这时候用「继续执行」就不行了，因为所有节点都已经完成了，系统会认为「没什么要执行的」。

#### 思考过程

1. **怎么定义「从这个节点重跑」？** → 重置这个节点 + 所有下游节点的状态，然后重新执行
2. **怎么找到下游节点？** → 递归遍历，从选中节点出发，沿着边的方向找所有可达节点
3. **上游节点怎么办？** → 保留已完成状态，不要重跑（结果还在）

#### 解决方案

- **新增「从此节点重跑」按钮**（红色）：只有选中节点时才显示
- **核心逻辑**：
  ```
  1. 找到选中节点的所有下游节点（递归）
  2. 重置这些节点的状态为 pending
  3. 清空这些节点的输出
  4. 调用执行引擎，从这些 pending 节点开始执行
  ```
- **与「继续执行」的区别**：
  - 「继续执行」（橙色）：跳过已完成节点，只跑未完成的
  - 「从此节点重跑」（红色）：重置选中节点及下游，重新跑

#### 效果

- 用户可以灵活地从任意节点开始重新执行
- 修改某个节点后，只需要重跑它和它的下游，不用从头开始

### 问题 6：独立任务（多棵树）被强制连接

#### 问题描述

最开始的 `validate_and_fix_graph` 函数有一个逻辑：如果发现孤立节点（没有任何边的节点），就把它强制连接到其他节点上。

但实际场景中，用户可能有多个完全独立的任务，比如：
- 任务 A：调研 Python 新特性
- 任务 B：调研 JavaScript 新特性

这两个任务之间没有依赖关系，就应该是两棵独立的树，并行执行。系统不应该强制把它们连起来。

#### 思考过程

1. **拓扑排序支持多棵树吗？** → 支持！Kahn 算法本身就从所有入度为 0 的节点开始，天然支持并行
2. **那为什么之前要强制连接？** → 可能是想避免用户不小心创建了孤立节点。但这个逻辑不对，应该把选择权交给用户
3. **需要检测吗？** → 检测但不强制，只给个提示：「检测到 N 个独立子图」

#### 解决方案

- **删除强制连接逻辑**：不再自动给孤立节点加边
- **只提示不干预**：检测到独立子图时，在日志里输出提示信息
- **更新系统提示词**：告诉 Plan Agent「允许多棵独立的子图」

#### 效果

- 用户可以创建多个独立任务，它们会并行执行
- 系统更灵活，不再替用户做「可能是错的」决策

### 问题 7：想用自然语言修改图，不想手动编辑每个节点

#### 问题描述

手动编辑工作流很繁琐：
- 想改 node_3 的指令 → 点节点 → 点编辑指令 → 输入 → 保存
- 想加个节点 → 点新增 → 选类型 → 输标题 → 保存
- 想删节点 → 点节点 → 点删除

能不能直接说：「把 node_3 的指令改详细点，然后加个节点在它后面分析结果」，让 AI 自动搞定？

#### 思考过程

1. **这需要什么能力？** → 需要一个能「理解修改指令 + 修改 DAG 结构」的 Agent
2. **从零做吗？** → 不用，复用 Plan Agent 的能力，只是输入从「任务描述」变成了「当前 DAG + 修改指令」
3. **前端要加什么？** → 一个「AI 修改图」的入口，让用户输入自然语言指令

#### 解决方案

- **新增 `modify_graph_stream` 接口**：接收当前 DAG + 修改指令，输出修改后的 DAG
- **新增系统提示词**：专门用于修改图的 Agent Prompt，告诉它可以做哪些修改
- **前端入口**：「🤖 AI 修改图」按钮（紫色），点击弹出输入框
- **使用示例**：
  - 「把 node_2 的指令改得更详细一些」
  - 「给 node_1 添加 web_search 工具」
  - 「在 node_3 后面加一个节点来分析结果」
  - 「删除 node_5，它没有必要」

#### 效果

- 简单的修改，说一句话就搞定了，不用点来点去
- 用户不需要记住节点 ID 也能操作（AI 会理解）

## 三、架构决策与考量

### 决策 1：自研执行引擎 vs 使用 LangGraph

| 维度 | 自研 | LangGraph |
|------|------|-----------|
| 灵活性 | 高，完全可控 | 中，受框架约束 |
| 学习成本 | 需要从零设计 | 需要学习框架 API |
| 生态 | 无 | 与 LangChain 生态深度集成 |
| 绑定程度 | 无，完全框架无关 | 深度绑定 LangChain |

**最终选择：自研**

原因：
1. 我们的核心定位就是「DAG 可视化 + 观测」，不想被任何执行框架绑定
2. 拓扑排序 + 并发调度本身不复杂，值得用少量代码换取完全的控制权
3. 以后如果想接 LangGraph，也可以作为一个「执行后端」接入，不影响整体架构

---

### 决策 2：纯观测平台 vs 规划+执行+观测一体化

| 维度 | 纯观测 | 一体化 |
|------|--------|--------|
| 复杂度 | 低 | 高 |
| 差异化 | 高（专注观测做到极致） | 中（跟 LangSmith 等正面竞争） |
| 落地难度 | 低（只需要接收事件） | 高（需要做好规划和执行） |
| 与 IDE 集成 | 容易（只需要挂在旁边看） | 难（需要接管执行） |

**最终选择：两条腿走路，但观测优先**

原因：
1. 观测是我们的独特优势，也是别人没做好的地方（DAG 可视化执行轨迹）
2. 规划和执行可以慢慢做，也可以接别人的（比如 Trae 的 Agent）
3. 「先看清楚，再优化」是更合理的路径

---

### 决策 3：嵌入 IDE vs 独立 Web 应用

| 维度 | IDE 插件 | 独立 Web |
|------|---------|---------|
| 使用场景 | 开发时随时看 | 任何地方都能看 |
| 部署成本 | 发布插件市场即可 | 需要服务器 |
| 数据隐私 | 本地存储，不用上传 | 需要考虑数据安全 |
| 与代码联动 | 强（可以看当前项目代码） | 弱 |

**最终选择：两者都支持，但插件优先**

原因：
1. 我们的目标用户是开发者，他们在 IDE 里的时间最长
2. Trae 基于 VSCode 内核，可以直接复用 VSCode 插件生态
3. 纯前端 + IndexedDB 可以做到零后端，发布插件后完全不用运维

## 四、经验与教训

### 4.1 流式输出不是「锦上添花」，是「必须有」

对于任何需要长时间运行的 AI 系统，**没有流式输出 = 黑盒 = 用户焦虑**。

哪怕只是简单地输出「正在处理...」「已完成 3/8 个节点」，也比让用户对着一个空白的加载图标强一万倍。

### 4.2 不要替用户做「可能是错的」决策

比如之前的「强制连接孤立节点」—— 系统觉得「这是为你好」，但实际上可能破坏了用户的真实意图。

更好的做法是：**检测 + 提示，把决策权还给用户**。

### 4.3 可视化是观测的核心，但不是全部

DAG 图很直观，但也需要配合：
- 时间线视图（线性顺序）
- 日志面板（详细输出）
- 统计面板（Token、耗时、调用次数）

不同的场景需要不同的视图。

### 4.4 自然语言交互是「效率放大器」

手动点 5 次才能完成的操作，如果说一句话就能搞定，用户体验的提升是巨大的。

AI 不应该只用来「从零生成」，更应该用来「修改和调整」—— 这是用户更频繁的需求。

## 结语

这个项目从最开始的「想让 Agent 执行过程透明一点」，慢慢演化成了现在的「DAG 规划 + 流式执行 + 轨迹观测」的完整系统。

一路走来遇到的这些问题，其实都围绕着同一个核心：**让用户对 AI Agent 的行为有掌控感和可预测性**。

AI 不是魔法，它的每一步决策、每一次工具调用、每一笔 Token 消耗，都应该被看见、被理解、被优化。

这就是我们做这个系统的初衷。
