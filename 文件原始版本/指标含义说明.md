# 指标含义说明

## RAG 检索指标

### recall_cap@5
**全称：** Recall Capped at 5
**含义：** 前 5 个检索结果中命中的相关文档数 ÷ min(5, 总相关文档数)。cap=5 表示分母最多取 5（如果相关问题数超过 5，分母就是 5；如果少于 5，分母按实际数算）。比普通 recall 更公平，不会因为一个问题关联了 50 篇文档就低估了模型在前 5 个结果中的表现。
**项目中使用：** rag.html、RAG检索优化.html

### recall@5
**全称：** Recall at 5
**含义：** 前 5 个检索结果中命中的相关文档数 ÷ 总相关文档数。分母是全部相关文档，如果一个问题有 50 篇相关文档，分母就是 50，即使前 5 个全中，recall@5 也只有 10%。
**项目中使用：** rag.html、RAG检索优化.html

### ndcg@5
**全称：** Normalized Discounted Cumulative Gain at 5
**含义：** 归一化折损累计增益。不是只看"有没有命中"，而是看命中的文档排得多靠前——越相关的排越前面得分越高，排到后面得分就折损。满分 1.0 意味着前 5 个结果不仅全相关，而且排序完全正确。
**项目中使用：** rag.html、RAG检索优化.html

### mrr@5
**全称：** Mean Reciprocal Rank at 5
**含义：** 平均倒数排名。只看"第一个相关结果排在第几位"——如果第一个相关结果排在第 1 位，贡献 1/1=1 分；排在第 3 位，贡献 1/3≈0.33 分；排在前 5 之外，贡献 0 分。衡量"用户是不是一眼就能看到想要的东西"。
**项目中使用：** rag.html、RAG检索优化.html

### precision@1
**全称：** Precision at 1
**含义：** 第一个检索结果是否相关。最严格的单一指标——只关心排在第一位的那个文档好不好。如果第一条就是用户想要的，precision@1=100%，否则=0%。常用于评估"搜索的第一条结果质量"。
**项目中使用：** rag.html、RAG检索优化.html

### f1@5
**全称：** F1 Score at 5
**含义：** 精确率与召回率的调和平均数（F1 = 2 × P × R / (P + R)）。综合衡量前 5 个结果的精确度（命中的是否都是相关的）和覆盖率（相关的是否都被命中到了）。越高越好，单一指标偏科会导致 F1 下降。
**项目中使用：** rag.html、RAG检索优化.html

### 端到端延迟
**含义：** 从用户发起查询到收到检索结果的完整耗时。RAG 项目中增加 Rerank 后延迟从 ~55ms 升至 ~360ms，带来约 3-5pp 的 recall_cap@5 提升。
**项目中使用：** rag.html、RAG检索优化.html

### P50 / P95 检索延迟
**含义：** P50（中位数延迟）表示一半请求在多少毫秒内完成；P95 表示 95% 的请求在多少毫秒内完成（排除最慢的 5%）。P95 更能反映系统的长尾性能。
**项目中使用：** rag.html（优化后 P50=380ms，P95=1.2s）

## 生成质量指标

### 忠实度（Faithfulness）
**含义：** 模型生成的回答是否严格基于检索到的文档内容，没有编造或歪曲事实。RAGAS 框架评分，越高表示生成内容越忠于原文。
**项目中使用：** rag.html（基线 72%，优化后 96%）

### 答案相关性（Answer Relevance）
**含义：** 生成的回答是否回答了用户的问题，有没有跑题或答非所问。RAGAS 框架评分。
**项目中使用：** rag.html（基线 65%，优化后 84%）

### 幻觉率（Hallucination Rate）
**含义：** 模型生成的内容中，与事实不符或没有文档依据的部分所占比例。越低越好。来源于优化后忠实度的反向推算（100% - 忠实度 ≈ 5%）。
**项目中使用：** 项目/index.html（18% → 5%）

## 效率/成本指标

### Token 消耗减少
**含义：** 优化后相比优化前每次查询平均消耗的 Token 数降低比例。通过问题改写、智能路由跳过不必要的检索等方式减少 Token 使用。
**项目中使用：** rag.html（↓ 42%）

### Embedding API 调用减少
**含义：** 通过智能路由（简单问题跳过向量检索）等方式，减少对 Embedding 服务的 API 调用次数。
**项目中使用：** rag.html（↓ 35%）

### 大文件上传响应速度提升
**含义：** 通过并发分片上传、Redis Bitmap 状态追踪、RocketMQ 异步解析等优化，大文件上传的响应速度提升百分比。
**项目中使用：** rag.html（↑ 65%）

### 页面初始化时间减少
**含义：** 通过 Caffeine+Redis 多级缓存、Pipeline 批处理等优化，社区平台页面首次加载的时间减少百分比。
**项目中使用：** community.html（↓ 60%）

### Token 消耗量（并行 vs 串行）
**含义：** Workflow 项目中，KAHN 算法并行调度相比串行执行减少的 Token 消耗比例。
**项目中使用：** workflow.html（↓ 14%）

### 并行执行时间（并行 vs 串行）
**含义：** Workflow 项目中，KAHN 算法并行调度相比串行执行节省的时间比例。
**项目中使用：** workflow.html（↓ 23%）

## Agent 任务指标

### DAG 图构建正确率
**含义：** Plan Agent 生成的 DAG 图（节点列表 + 边关系）在结构上是否正确（无环路、无缺失依赖、节点间逻辑合理）的比例。
**项目中使用：** workflow.html（97%）

### 子 Agent 任务完成率
**含义：** 子 Agent（调研/审查/安全等角色）独立完成自身节点任务的成功率，包含正确调用工具、产出符合预期的输出。
**项目中使用：** workflow.html（94%）

## 长期记忆指标

### Information Extraction（精确检索）
**含义：** 从对话历史中精确提取特定事实信息的能力。例如"用户上个月提到了喜欢的电影是什么"——模型需要从大量历史记录中准确找到答案。
**项目中使用：** myclaw.html（Hindsight 95%）

### Multi-Session Reasoning（跨会话推理）
**含义：** 跨越多个对话会话进行逻辑推理的能力。例如"用户第一轮说了喜欢猫，第二轮说了养了一只布偶，那用户喜欢什么品种的猫？"
**项目中使用：** myclaw.html（Hindsight 90%）

### Knowledge Updates（知识更新）
**含义：** 当用户的信息发生变化时（如"之前喜欢 A，现在改喜欢 B"），模型能否正确追踪并更新记忆，而不是停留在旧信息上。
**项目中使用：** myclaw.html（Hindsight 88%）

### Temporal Reasoning（时序推理）
**含义：** 基于时间顺序进行推理的能力。例如"用户上个月说了什么，这周又说了什么，两者有什么关联？"
**项目中使用：** myclaw.html（Hindsight 90%）

## 其他指标

### 告警频率限制
**含义：** 异常邮件告警的防重复机制，同一异常在 10 分钟内只发送 1 次告警邮件，避免告警风暴。
**项目中使用：** community.html

### 限流（Token Bucket）
**含义：** MyClaw 的 NetworkPolicy 模块使用令牌桶算法限制外部 API 调用频率，每分钟最多 30 次。
**项目中使用：** myclaw.html
