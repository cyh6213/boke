RAG 智能路由：八种查询类型的检索策略与优化

AI RAG 学习笔记
作为企业知识库 Agentic RAG（项目01）的补充扩展，本文系统梳理了我在企业知识库 Agentic RAG 项目中对各类查询的优化思路、实现过程，以及与传统 RAG、GraphRAG 的能力对比分析。
引言 为什么需要为查询类型定制策略

我把用户查询分成了 8 种类型，每种类型在检测方式、检索策略、生成策略三个维度上有本质区别：

#类型例子需要特殊处理的点
⓪无需检索"你好"，"1+1="基于规则引擎直接过滤，走模型回答
①单跳事实查询"程愚瀚学什么专业？"高阈值精确召回，严格限定来源
②对比查询"A 手机和 B 手机性能对比"需要两次独立检索，分成 A/B 池
③技术操作查询"怎么配置 Spring Security JWT？"找步骤类文档，分步骤模板输出
④时序与时效性查询"今年 Q3 营收和去年比？"/"最近有什么新功能？"需要时间元数据 + 时间过滤 / 新鲜度排序
⑤验证查证查询"有人说这个产品召回率 90%，对吗？"声明抽取 → 事实检索 → 比对
⑥列举聚合查询"我们公司有哪些项目？"多次检索 + 去重合并
⑦多跳推理查询"程愚瀚和鲁迅是什么关系？"子查询链 + ReAct，逐步搭桥

前 7 种每一种需要的改动都不大，但效果提升显著。第 8 种（多跳推理）是改动最复杂、也是本文重点阐述的优化方向。

通用层 三层路由检测（所有查询的入口）

所有查询在进入 7 种类型策略之前，先经过三层递进检测，确定问题类型和复杂度：

第一层 · 规则引擎（O(1) 快速过滤） 
匹配模式：
- 闲聊类：「你好」「今天天气」→ 直接模型回答
- 纯计算类：「1+1=」→ 直接模型回答
- 已知无结果类：「推荐一部电影」→ 根据系统配置判断

成本：零模型调用，基于关键词 + 正则匹配，毫秒级过滤。

第二层 · 语义分类器（LLM 判断维度） 
基于 Spring AI Alibaba 的 QuestionClassifierNode，从 4 个正交维度独立判定：

维度 A · 检索模式（决定怎么搜）：
  none → 无需检索（闲聊/计算类）
  single → 单次精确检索（单跳/操作/验证）
  dual_pool → 双池独立检索（对比查询）
  multi_retrieve → 多次检索+补充（列举聚合）
  multi_hop → 多次检索+ReAct搭桥（多跳推理）

维度 B · 时间要求（决定时间过滤与排序）：
  none → 无时间要求
  time_range_or_freshness → 指定时间段或要最新的（时序/时效性查询）

维度 C · 生成格式（决定输出模板）：
  fact → 事实陈述+来源
  step → 分步骤模板
  comparison → 按维度对比表格
  list → 列表+分类汇总
  chain → 推理链可视化
  verify → 声明vs事实比对

维度 D · 验证标记（是否需验证用户提供的声明）：
  false → 正常查询
  true → 需要验证

成本：一次小模型分类调用，约 50ms。4 个维度各自独立输出，自然支持类型共存（如对比查询+时序查询同时标记）。

第三层 · 实体关系检测（量化复杂度评分） 
评分公式：complexity_score = n_entities * w1 + n_relations * w2 + depth_score * w3

其中：
- n_entities：提取出的候选实体数量
- n_relations：实体间的二元关系对数
- depth_score：推理深度估计（如"A→B→C"为 2 跳）
- w1, w2, w3：可调权重

if complexity_score > threshold → 判定为多跳查询

成本：纯规则计算，可以做到不加额外延迟。

经过三层判定后，查询携带 4 个维度的标注值（检索模式/时间要求/生成格式/验证标记），被分配给对应的策略组合执行。各维度独立取值，自然支持类型共存。

2.0 无需检索

定义：不需要从知识库检索即可直接回答的问题，如问候、纯计算、以及系统配置为"不回答"的已知无结果类问题。

判断方式：由第一层规则引擎直接识别，不走语义分类器和向量检索。匹配模式包括：

- 闲聊类：「你好」「今天天气」→ 直接模型回答
- 纯计算类：「1+1=」→ 直接模型回答
- 已拦截类：「推荐一部电影」「我的密码是什么」→ 根据系统配置判断，返回预设回复
检索策略：跳过向量检索和 Embedding 调用，不走检索管线。
生成策略：闲聊和计算类由 LLM 直接回答；已拦截类返回配置好的固定回复模板。

目的：降低 Embedding API 调用成本，提升响应速度。

测试数据：
方案Embedding 调用P50 延迟P95 延迟
优化前（无路由，统一走全链路）1 次/查询620ms2.1s
优化后（规则引擎直接拦截）0 次/查询380ms1.2s

优化效果：Embedding API 调用减少 35%，P50 延迟降低 39%，平均 Token 消耗降低 42%。

优化维度总览

接下来 8 种类型的优化都围绕以下 3 个维度展开，每种类型只是在这 3 个维度上做了不同的选择：

优化维度可调整的选项影响
检索策略检索模式：single 单次(1检索) / dual_pool 双池独立(2检索) / multi_retrieve 多轮补充(2-3检索+去重) / multi_hop ReAct链(N检索+搭桥)决定延迟和成本基数
阈值与候选池：高阈值0.75精确(top-10) / 低阈值0.6宽召回(top-30) / 多轮补充(top-50+补充)权衡精确率 vs 召回率
改写策略口语化 / 关键词提取 / 多角度扩写 / 后退提问 / 声明抽取影响检索命中率
生成策略严格限定 / 分步骤模板 / 对比格式 / 时序标注 / 声明比对决定输出质量和幻觉率

前 6 种类型主要通过调整这 3 个维度的参数实现优化，第 7 种（多跳推理）除此之外还需要额外的执行引擎。

2.1 单跳事实查询

定义：一次检索即可回答的事实性问题，答案集中在一个文档片段内。

检索策略：检索次数 1 次，检索阈 值 0.75（高精确），候选数 top-10 → Rerank top-3，改写策略仅做口语化处理和指代消除。
生成策略：严格限定在召回内容内，每个结论必须标注来源，禁止引入 LLM 外部知识。
测试数据：
方案recall_cap@5P50 延迟
优化前（无路由，统一走全链路）68.4%620ms
优化后（路由判定+高阈值精确检索）81.2%360ms

优化效果：路由判定为单跳事实后使用高阈值精确检索，召回率提升 +12.8pp，延迟降低 42%。

2.2 对比查询

定义：需要比较两个或多个实体的差异，如"A 和 B 的性能对比"。

检索策略的特殊性：检索次数 2 次独立检索，分 A 池和 B 池；各池独立设定高阈值（0.75）；改写策略分别抽取 A 和 B 的检索词，隔离检索防止互相污染。
生成策略：按维度对比输出（如性能、价格、适用场景等维度）；A 池和 B 池的结果各自标注；单边缺失：如果 A 池搜到了 B 池没搜到，标注"关于 B 的信息不足"；禁止用 LLM 内部知识填补缺失方的内容。
测试数据（200 条对比查询）：
优化完整对比率单边幻觉率
无优化（单次检索+LLM 硬比）52%38%
双池独立检索78%11%
+ 单边缺失标注78%3%
2.3 技术操作查询

定义：询问如何完成某项操作，格式通常为"怎么/如何/步骤"。

检索策略的特殊性：检索次数 1 次；改写策略提取操作关键词，去掉疑问词，转成"操作说明"风格的查询；检索时优先匹配包含步骤编号、代码块、配置列表的文档；可以在文档元数据中标记"步骤类文档"提高命中。
生成策略：分步骤模板输出，每步带引用；如果包含代码，用代码块展示；如果没有找到完整步骤，告知"找到的信息不完整"并给出已有片段。
测试数据（100 条操作查询）：
优化步骤完整性代码可运行率用户满意度
无优化（通用检索）58%42%61%
+ 操作词提取改写72%55%74%
+ 步骤标签匹配83%68%85%
2.4 时序与时效性查询

定义：涉及时间维度的查询，包含两种情况——指定时间点/段（"去年Q3的营收"）和要最新的（"最近有什么新功能发布"）。两者在业界分类中常被拆成 Temporal 和 Hot，但在实际查询中边界模糊（"最近"和"上个月"本质都是时间约束），因此合并为同一类，通过不同的时间参数区分。

检索策略的特殊性：需要文档携带时间元数据；检索时注入时间参数，两种模式——time_range：指定时间范围过滤；freshness：按时间倒序排序，优先最新文档；同类型多时间点对比时走双池策略；排序策略也会根据模式调整。
生成策略：每个数据点标注时间戳；趋势类问题要求输出时间线；如果某个时间点没有数据，明确标注"在 X 时间范围内未找到记录"；时效性查询要求标注"信息截止时间"。
测试数据（120 条，含 60 条时序 + 60 条时效性）：
优化时间准确率新鲜度准确率数据完整率延迟影响
无优化（语义搜索硬推理）34%28%28%+0ms
+ 时间元数据 + 时间参数注入82%86%71%+15ms
+ 分模式排序策略86%91%73%+20ms
2.5 验证查证查询

定义：用户看到某个信息，来知识库确认是否准确。

检索策略的特殊性：先做声明抽取：从用户问题中提取出"需要验证的事实"；再将声明转成确定性查询去检索；如果检索到的结果与声明相符 → 验证通过；如果不相符 → 给出实际数据，标注差异。
生成策略：声明 vs 事实的对比展示；差异高亮：哪些对、哪些不对、哪些查不到；查不到的内容要明确说"在知识库中未找到相关信息"。
测试数据（80 条验证查询）：
优化验证正确率误判率拒绝率
无优化（直接把问题当查询）41%33%0%
+ 声明抽取 + 精准检索76%11%13%
2.6 列举聚合查询

定义：需要返回满足条件的多个结果，而不是单一答案。

检索策略的特殊性：不是找"最相似的文本"，而是找满足条件的全部内容；需要扩大候选池：低阈值（0.6）宽召回，候选集 top-50；多轮检索：如果第一轮结果数不够，用第一批结果中的关键词补充检索；去重合并。
生成策略：列表式输出，每个条目标注来源；分类汇总：按类型/时间/主题等维度分组；当结果超过 10 条时，提供简要版和完整版；标注"共找到 X 条结果"。
测试数据（60 条列举查询）：
优化查全率结果数中位数延迟
无优化（top-5 高阈值）31%2180ms
+ 低阈值宽召回58%8420ms
+ 多轮补充检索76%14850ms
2.7 多跳推理查询

多跳推理是八种类型中实现最复杂、优化最深的一类。它不像前面七种只需要调整检索参数或做简单的策略分支，而是需要一套完整的执行引擎（检测 → 分解 → 逐跳检索 → 推理链组装 → 生成）。下面详细记录完整的优化过程。（检测 → 分解 → 逐跳检索 → 推理链组装 → 生成）。下面详细记录完整的优化过程。

为什么需要多跳推理

在做这个之前我首先确认了一个问题：真实场景中的多跳查询到底有多"多跳"？

我对知识库中的用户查询做了一个统计，按查询需要跳转的次数分布如下：

跳数占比说明
0 跳（无需检索）~15%闲聊、计算类，规则引擎直接过滤
1 跳（单次检索）~60%"程愚瀚学什么专业"——一次检索搞定
2 跳~15%"程愚瀚和鲁迅的关系"——需要搭一段桥
3 跳~7%需要跨两个中间实体
4 跳及以上~3%长链推理，非常罕见

从这个分布可以看得很清楚：

1. 95% 以上的查询在 2 跳以内。绝大多数用户问题不需要复杂的长链推理。

2. 我的方案刚好覆盖了这个需求区间。2-3 跳的逐次检索可以很好地解决。

3. GraphRAG 在这个区间并不划算。以 HotPotQA 的多跳基准测试为例——微软 GraphRAG 建图成本约 $100-200 / 100 万 token，而我的方案额外建图成本为 $0（沿用已有向量索引）。在 2-3 跳场景下，我的方案预期准确率约 65-80%，GraphRAG 约 60-70%，差距不大。

结论

多跳推理要解决的问题就是卡在单次向量检索和 GraphRAG 之间的那部分——单次 RAG 搞不定（一次检索找不到关联），GraphRAG 又太贵太重（建图成本高、更新麻烦）。这套方案就是用 2-3 次向量检索 + ReAct 搭桥，以极低的额外成本覆盖了这约 25% 的中等复杂查询。

2.7.3 子查询链 + ReAct 执行

以"程愚瀚和鲁迅的关系"为例，完整的执行过程：

Round 1:
  Query: "程愚瀚和鲁迅的关系是什么？"
  ├── 问题分解 → 子问题①："程愚瀚是谁？"
  ├── 混合检索（dense+sparse, th=0.6）
  ├── Rerank 重排
  └── 检索结果："程愚瀚，电子科技大学软件工程 28 届本科生"
      → 中间结论：程愚瀚 → 电子科技大学

Round 2:
  Query: "电子科技大学培养了什么知名人物？"
  ├── 基于结论①构建
  ├── 混合检索（dense+sparse, th=0.6）
  ├── Rerank 重排
  └── 检索结果："电子科技大学历史上培养了许多杰出人才..."
      → 中间结论：电子科技大学 → 鲁迅（间接关联）

Round 3（推理链闭合）:
  合并结论① + 结论②：
  "程愚瀚是电子科技大学学生，电子科技大学的历史人才中
   有与鲁迅相关的人物，因此程愚瀚与鲁迅存在间接关联"
      → 最终结论：程愚瀚 → 电子科大 → 鲁迅（间接关联）

① 结构化输出 — 每步结论都标注来源

关键发现
发现一：单跳 + 小库 → 传统 RAG；多跳 + 小库 → 定制策略 RAG；多跳 + 大库 → GraphRAG。
发现二：你的方案在中等规模知识库（10w-50w 文档）上可能最优，在不建图的前提下达到接近 GraphRAG 的效果。
总结 核心结论与下一步
核心结论
1. RAG 系统必须按查询类型定制策略。7 种查询类型的检索和生成需求差异巨大，一刀切的方案要么成本过高，要么效果不够。
2. 多跳推理是 RAG 系统能力的真正分水岭。前 6 种类型只需要调整检索参数，而多跳推理需要一套完整的执行引擎。
3. 你的"动态推理链"与 GraphRAG 的"静态层次图"是两条差异化的技术路径。你的方案在中小型知识库更具优势。
