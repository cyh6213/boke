Agent Token 优化实践

2026 年 7 月 2 日 · Agent Token 性能优化

我的项目中的 token 节省实战经验。

Token 构成占比

很多人提到 Token 优化时从"用户的输入"入手，但这是一个误区。用户输入在一次调用中的占比其实相对较小。一次完整的 Agent 调用，Token 构成大概是（以 Workflow 项目为例）：

组成部分大致占比说明
System Prompt（角色+规则+工具）50%-60%固定的，每次调用都注入
上下文/历史20%-30%随轮次增长
工具返回结果10%-15%取决于工具的输出大小
用户输入< 5%占比最小

用户在每次提的问题通常就几十到几百 Token，在这上面做优化的空间很有限。而 System Prompt 占了超过一半，上下文和工具结果加起来也占了三四成——这两块才是优化的主战场。

所以下面的部分对应占比最大的几个组成部分：System Prompt、上下文、工具描述、API 调用。架构层面单独成章，它对应的不是某一块 Token 占比，而是通过减少执行轮次、限制上下文传播范围等机制影响上述所有组成部分的整体消耗。

一、System Prompt 优化

占比分析

System Prompt 是每次调用 Token 消耗中占比最大且最容易被忽视的部分。以 MyClaw 项目为例，一个完整的 Worker Agent 的 System Prompt 有 11 层，仅其核心规则部分就占了可观的 Token 量。

双 Agent 差异化加载

在使用 OpenClaw 时我发现一个问题：即使我只是跟它随意聊天，它的 System Prompt 也始终包含完整的工具列表、策略配置、执行规则等重内容。每次对话都在为"根本用不到的能力"支付 Token 成本。

所以在 MyClaw 项目中，我设计了 Worker + Chatter 双 Agent 架构，让两者的 System Prompt 不同：

  • Chatter（聊天 Agent） — 只带 5 层精简上下文：身份设定、语言规则、运行时上下文、记忆检索、聊天规则。不加载工具列表、策略配置、审批规则等重内容。相当于只保留"聊天必需"的部分。
  • Worker（后台 Agent） — 带完整的 11 层，除 Chatter 的 5 层外，额外加载工具列表、策略文件、权限规则、Skills 索引、上下文管理规则等。

这样做的优势很明显：Chatter 只负责对话时，不需要知道 Worker 的工具和策略，省去了每次对话都要注入工具列表和策略描述的 Token。这是通过职责分离实现 Token 优化的一种体现。

规则分级与精简（Token 审计）

为了进一步节省 Token，我又对 Workflow 项目的 System Prompt 做了一次完整的 Token 审计——把每次会话都会注入的规则列出来，看看哪些真是每次都需要、哪些大部分时候用不上。

审计时发现一个典型问题：有一些流程规则（比如"多步骤依赖处理"）在每一个 SubAgent 的 Prompt 中都注入了一遍，但实际上很多 SubAgent 负责的内容根本跟多步骤没关系——「调研 Agent」只需要搜索和阅读文件，"多步骤依赖"的规则对它完全是冗余的。

优化方式很简单：把规则分成两级——

  • L1 核心规则 — 必须每次加载的，比如安全拦截、格式约束。数量少，但每次都带上
  • L2 场景规则 — 绑定到 SubAgent 角色上，Plan Agent 分配节点时自动注入。

审计前 Always-Apply 规则总共约 11K Token，优化后精简到 5.8K，基础开销降低 36%。省出来的空间可以让 32K 窗口的有效可用容量从 8.5K 提升到 17K。

二、上下文管理（短期记忆）

业界做法

Claude Code 的多层渐进式压缩： Claude Code 的上下文管理是我认为设计得最完善的一套方案。它采用了三级渐进式压缩策略，从轻到重逐级升级：

  1. MicroCompact（微压缩） — 零成本操作。利用 Prompt Cache 直接删除旧的工具调用结果，或对超过 60 分钟的旧消息做占位替换。核心原则：读文件/执行命令/搜索类结果可以压缩，但用户消息、Agent 分析文本、系统消息必须保留。
  2. SessionMemoryCompact（会话记忆压缩） — "以空间换时间"。后台进程在每轮对话后自动提取关键信息写入记忆文件，压缩时直接用这份摘要，不需要临时调 API。
  3. Full Compact（全量压缩） — 最后手段。调用 API 生成一份九段式结构化摘要替代旧消息，包括用户意图、核心概念、涉及文件、遇到的错误、待办任务等。为了保证压缩时不调用工具，API 请求前加一段强硬声明禁止工具调用。

OpenClaw 的 QMD 记忆后端： OpenClaw 通过 QMD 记忆后端实现了 BM25 关键词匹配 + 向量语义搜索 + LLM 重排序的三层混合检索。核心思路是不再把整个记忆文件塞进上下文，而是只提取最相关的 2-3 句片段传给模型。据实测，长会话从 80000 tokens 削减 95% 以上。

Workflow 中的做法

Workflow 项目目前在这块的实现比较简单——节点粒度上下文隔离。每个 SubAgent 在拿到节点时，只获得当前执行所需的信息：

  • 上游节点的输出摘要（不是完整执行过程）
  • 当前节点的任务描述（要做什么、输入参数是什么）

不获得的信息包括：上游节点的完整执行历史、全局 DAG 图、其他 SubAgent 的中间状态。

对比 LangGraph 的全局 state 方案（每次保存完整快照），DAG 图分开存储的方式天然更节省 Token，因为我们只传递"当前节点需要知道的内容"。

三、工具调用优化

问题：工具描述膨胀

一个 Agent 如果绑定了 10 个 Tool，每个 Tool 的描述 + 参数 JSON Schema 加起来可能有几千 Token。如果所有 Tool 的描述每次都全量注入，即使是当前用不到的 Tool 也在白白消耗 Token。

按 SubAgent 身份绑定工具（Workflow）

这里的做法是：不按全局绑定工具，而是按 SubAgent 身份绑定。每个 SubAgent 有自己的工具列表：

  • 调研 Agent — 只绑搜索类工具（WebSearch、文件读取）
  • 审查 Agent — 只绑代码检查类工具
  • 执行 Agent — 只绑执行命令/写文件类工具

同时划分公共工具池和私有工具池：

  • 公共工具（如基础文件读写）— 所有 SubAgent 共享
  • 私有工具（如特定 API 调用）— 仅特定 SubAgent 可用

这样每个 SubAgent 的 Prompt 中只注入跟自己相关的工具描述，而不是全部工具列表。对比一把梭的方式，每次调用的工具描述部分能省下可观的 Token。

四、用户输入

用户输入在一次调用中的占比通常不到 5%，所以不需要在这里花太多精力去优化。Token 优化的重心应该放在 System Prompt 和上下文管理上，而不是纠结用户的输入长度。

五、架构层面（Workflow）

为什么单独加这部分

乍一看架构层面的 SubAgent 好像跟前面三个部分有重复。确实，SubAgent 的 Token 节省本质上就是前面三部分的体现：

  • System Prompt 优化 → SubAgent 只需要更少的 System Prompt（5 层 vs ReAct 的 9-10 层）
  • 上下文管理 → SubAgent 只获得前一个节点的输出，不背整段对话历史
  • 工具调用优化 → SubAgent 只绑自己角色需要的工具和 Skill，不是全部

但这三个优化在单 Agent 方案里也能做（比如 MyClaw 的 Chatter 只带 5 层），为什么还需要单独的架构层？区别在于：这些优化在单 Agent 里是"减负"，在 Workflow 里是"天生如此"。

单 Agent 方案里，你做一个轻量的 Prompt，本质是在同一个 Agent 身上做减法——你得手动判断"这个场景下哪些内容可以去掉"。而 Workflow 通过职责拆分，让每个组件天然只需要承担一小部分能力，System Prompt 轻量不是刻意优化的结果，而是架构设计带来的副产品。不需要判断"这个场景要加载什么规则"——因为 SubAgent 本身就只做特定场景的事。

System Prompt 构成对比

Workflow 的 System Prompt 设计是 Token 节省的关键。跟"一个 Agent 用 ReAct 执行所有任务"的方案对比，每一层的差距都很明显：

对比项单 Agent ReAct（基线）Workflow Plan AgentWorkflow SubAgent
角色定义~1K（完整人格+灵魂+身份）~0.2K（仅"你是规划 Agent"）~0.2K（仅"你是 xx Agent"）
工具列表~4K（全部工具描述+Schema）无（不调用工具）~1K（只绑当前角色用的 2-3 个）
策略/规则~2K（审批、模式、权限、Skills）~0.8K（拆解规则+颗粒度规则）~0.8K（L1 基础规则+L2 角色规则）
记忆/上下文~2K（记忆检索+历史对话摘要）~0.5K（可用角色清单）~0.5K（当前任务输入）
FewShot无或全部~1K（DAG 反例样本）无
单次调用总计~9-10K（随轮次增长）~2.5-3K（固定）~2.5-3K（固定）

差距源于三者 System Prompt 的构成完全不同：

单 Agent ReAct（基线）的 System Prompt 构成：

1. 角色身份 + 人格灵魂（~1K）
2. 所有工具列表 + 参数 Schema（~4K）
3. 策略规则 + 审批流程 + 模式指令（~2K）
4. 记忆检索 + 历史上下文（~2K）
5. 运行时信息（日期/时间）
→ 全能但臃肿，每次调用全部加载

Plan Agent 的 System Prompt 构成：

1. 角色定义 — "你是任务规划 Agent"（~0.2K）
2. 可用 SubAgent 角色清单 — 有什么类型的执行单元、各自能做什么（~0.5K）
3. 拆解规则 — 怎么切分任务、怎么定依赖关系（~0.5K）
4. 颗粒度规则 — 什么粒度算一个节点（~0.3K）
5. 输出格式约束 — DAG 的 JSON Schema（~0.5K）
6. 反例样本 — 错误 DAG + 修正（~1K）
→ 只做规划，不涉及执行，不需要工具描述和策略规则

SubAgent 的 System Prompt 构成：

1. 角色定义 — "你是 xx Agent"（~0.2K）
2. L1 基础规则 — 安全拦截、输出格式、执行规范（~0.5K）
3. L2 角色规则 — 专属规范和流程（~0.5K）
4. 工具列表 — 只绑当前角色需要的 2-3 个工具（~1K）
5. 当前任务输入 — 上游输出摘要 + 节点描述（~0.5K）
→ 只做执行，不关心全局，不需要人格和记忆

调用轮次的差异

以一个 3 步任务为例：

单 Agent ReAct：

第1轮：System(9K) + 用户需求(0.2K) → 输出 → 上下文变成 10K
第2轮：System(9K) + 上轮历史(10K) + 新结果 → 输出 → 上下文变成 20K
第3轮：System(9K) + 上轮历史(20K) + 新结果 → 输出 → 上下文变成 30K
三轮总消耗：9+10+20+30 = ~69K（历史持续增长）

Workflow：

第1轮：Plan Agent(3K) + 用户需求(0.2K) → 输出 DAG

第2轮：SubAgent A（节点内部也有多步循环）
  第2-1步：System(3K) + 节点描述(0.3K) → 输出 → 上下文变 3.5K
  第2-2步：System(3K) + 上步历史(3.5K) → 输出 → 上下文变 6.5K
  第2-3步：System(3K) + 上步历史(6.5K) → 输出 → 节点完成
  节点内部消耗：3.3 + 6.5 + 9.5 = ~19K

第3轮：SubAgent B（不继承 A 的历史，重新开始）
  第3-1步：System(3K) + 节点描述(0.3K) → 输出 → 上下文变 3.5K
  ...
  节点内部消耗类似，~19K

三轮总消耗：3.2（Plan）+ 19（SubAgent A）+ 19（SubAgent B）= ~41K

差距原因不是"SubAgent 不会累计上下文"——它内部也会循环，也会增长。真正的差距在两点：

  1. System Prompt 更轻（3K vs 9K），每步的基数就小
  2. 节点间上下文重置——SubAgent B 不会带着 SubAgent A 的历史对话重新开始，每次节点结束历史清零。而 ReAct Agent 的历史会贯穿整个任务，越到后面越重

单次调用： 3K vs 9K，降低约 67%
完整任务（3 步）： 41K vs 69K，降低约 41%（主要来自 System Prompt 更轻 + 节点间上下文重置阻止了历史持续膨胀）

六、API 调用层面的成本优化

问题分类 + 智能路由

RAG 项目中实现了一套基于问题分类的智能路由机制。核心思路是：不是所有问题都需要走完整的 RAG 检索链路。

具体做法：

  • 对用户输入的问题先做分类判断，识别问题类型：问候、数学计算、常识问答、需要检索的问题等
  • 简单问题（问候、数学等）通过关键词/正则匹配直接回答，不调用 Embedding 模型做向量化，不执行检索
  • 复杂问题（需要查知识库的）才走完整链路：问题改写 → 向量化 → 双路检索 → RRF 融合 → Rerank 重排

这样做的成本节省有两方面：

  • Embedding API 调用减少约 35% — 简单问题不需要向量化，直接省掉了这部分 API 成本
  • LLM 生成质量提升 — 跳过不必要的检索，避免检索到的无关内容干扰模型生成

这虽然不是直接的 Token 节省，但减少了不必要的 API 调用次数，是成本优化的重要手段。

七、未实现的想法

内置命令跳过 LLM

对于流程固定的机械操作（如格式转换、数据校验），目前即使任务很简单，也必须经过 LLM 处理才能执行。一个更好的做法是：在注册工具时加上 skip_llm: true 标记，执行引擎遇到这类任务时直接调用对应的函数处理，不走 LLM。

判断标准不是"任务高频"，而是"这个任务有没有确定的输入输出转换规则"。如果有，就可以用代码替代模型推理。

代码执行替代工具循环

Anthropic 提出过一个思路：不是让 LLM 反复调工具传数据，而是让 LLM 写一段代码去执行多步操作。比如要处理一个大型文档，传统方式是 LLM 调"读取"工具拿到全文，再调"分析"工具传回去——文档内容在两次调用来回中两次经过模型上下文。更好的做法是让 LLM 写一段脚本，在沙箱里一次性完成读取+分析，只把最终结果返回给模型。

核心思路：把"LLM 驱动工具"变成"LLM 编写脚本"，减少中间结果反复穿过模型上下文的浪费。

模型分级路由

不是所有任务都需要最强的模型。简单任务（格式校验、简单分类）可以用小模型处理，只有复杂推理（代码审查、报告生成）才调用大模型。目前所有 SubAgent 都用同一个模型，如果按任务复杂度分级路由，在保证效果的前提下还能进一步降低成本。

语义缓存

对于用户重复提出的类似问题，可以用向量检索判断语义相似度，命中后直接返回缓存的回答，不走 LLM。在 RAG 项目的客服场景中，语义缓存的命中率可以达到 35% 左右，能进一步降低 LLM 调用次数。

Prompt 压缩

工具如 LLMLingua 可以自动压缩 Prompt，在保留核心信息的前提下将 Token 量压缩 5-20 倍。对于 SubAgent 的输入（如上游节点的输出摘要），如果摘要较长，可以先做一次无损压缩再注入上下文。
