Agent Token 优化实践

2026 年 7 月 2 日 · Agent Token 性能优化

很多人提到 Token 优化时，第一反应是"让用户少说点话"或者"精简 Prompt 措辞"。但这是最大的误区——用户输入在一次调用中的占比通常不到 5%，真正的优化空间在别的地方。

要优化 Token，首先得搞清楚一个 Agent 每次调用时，上下文到底由哪些部分组成。

一、Agent 上下文的构成

一个 Agent 每次调用 LLM 时，发送给模型的上下文并不是简单的一段文字，而是由多个独立部分拼接而成。以 Claude Code 为例，它的上下文由以下六层组成：

层级内容典型 Token 数说明
工具定义所有可用工具的 JSON Schema 描述~17,600最大的单一消耗者
System Prompt身份定义、行为规则、任务规范~2,600核心指令，决定 Agent 的行为框架
记忆/上下文CLAUDE.md、项目规范、记忆文件~1,000 ~ 10,000持久化知识，每次会话加载
对话历史 + 工具结果历史轮次 + 文件内容、Shell 输出等30% ~ 40%随轮次持续增长
用户输入当前轮次的用户查询< 5%占比最小，无需优化

可以看到，工具定义是最大的单一消耗者，其次是对话历史。用户输入占比最小，在这上面花精力优化是得不偿失的。下面我按三个主要优化方向来展开：工具定义层、对话历史层、架构层面。

二、工具定义层优化（最大头）

问题

一个 Agent 如果绑定了 10 个 Tool，每个 Tool 的描述 + 参数 JSON Schema 加起来可能有几千 Token。如果所有 Tool 的描述每次都全量注入，即使是当前用不到的 Tool 也在白白消耗 Token。以 Claude Code 为例，27 个内置工具的 Schema 就占了约 17.6K Token——这是上下文中最大的一块。

方案：按 SubAgent 身份绑定工具

我的做法是：不按全局绑定工具，而是按 SubAgent 身份绑定。每个 SubAgent 有自己的工具列表：

  • 调研 Agent — 只绑搜索类工具（WebSearch、文件读取），不绑执行命令和写文件工具
  • 审查 Agent — 只绑代码检查类工具，不绑搜索和执行工具
  • 执行 Agent — 只绑执行命令/写文件类工具，不绑搜索工具

同时划分公共工具池和私有工具池：

  • 公共工具（如基础文件读写）— 所有 SubAgent 共享
  • 私有工具（如特定 API 调用）— 仅特定 SubAgent 可用

这样每个 SubAgent 的 Prompt 中只注入跟自己相关的工具描述，而不是全部工具列表。对比一把梭的方式，每次调用的工具描述部分能省下可观的 Token。

效果：单 Agent ReAct 一次调用加载所有工具的 System Prompt 约 9K，Sub Agent 只加载自己相关的约 3K，单次调用降低约 67%。省下来的主要是工具定义这部分。

为什么 Claude Code 的 Dynamic Workflow 不能这样做？因为 Claude Code 的 Harness 是通用基础设施——每个 agent() 可能执行任何操作（读文件、写代码、运行命令），你无法预知子 Agent 需要什么工具，所以必须加载全部 27 个工具的 Schema。而我的项目是专用系统，Plan Agent 生成的 DAG 明确知道每个 Sub Agent 需要什么工具，所以可以精确裁剪。这是领域专用性带来的优势。

三、对话历史层优化

问题

在 DAG 图中，节点之间有明确的先后依赖关系，后面的节点需要拿到前面节点的输出才能执行。原本的做法是把每个子 Agent 的完整对话历史——包括 system prompt、工具调用、工具返回结果、中间推理过程——全部存下来，然后直接传给下一个节点。

这样会带来两个问题：

  • Context 膨胀：一个子 Agent 执行一次任务可能产生多次工具调用，完整对话历史轻松达到 10K-50K+ tokens。多个节点串联后，下游 Agent 的 context 窗口很快被占满。
  • 信息干扰：大量无关信息会干扰 Agent 的判断，导致下游节点解析效率下降、准确率降低。

方案：动态 Schema 格式化输出 + 压缩兜底

我考虑了两种策略：

策略一：压缩策略（借鉴 Claude Code） — 子 Agent 执行完后，把完整的对话历史压缩成摘要，再传给下一个节点。好处是不需要改子 Agent 的行为，直接在传递层加一道压缩就行。但问题在于压缩本身也是一次模型调用，会额外消耗 token，而且压缩后的摘要可能会丢失细节。

策略二：动态 Schema 格式化输出（借鉴 Dynamic Workflow） — 在子 Agent 的 prompt 里明确要求它只输出结构化的结果，而且 schema 是动态的——Plan Agent 在生成 DAG 图的时候，会同时给每个节点生成一个 output_schema 字段。比如查天气的节点 schema 是 {city, temperature, condition}，生成报告的节点 schema 是 {advice, warning_level}。执行引擎在启动子 Agent 时把这个 schema 注入到 prompt 里，子 Agent 就知道自己该输出什么结构。

权衡之后我选择了第二种策略，因为它更根本——不是"把脏数据洗干净再传"，而是"一开始就不产生脏数据"。压缩策略在长文本场景下作为兜底手段。

效果：格式化输出砍掉了子 Agent 推理过程中 60% 到 80% 的冗余 token，因为大部分对话历史里工具调用和中间思考占了绝大部分。动态 schema 的方式让每个节点的输出结构都是任务特定的，下游节点可以直接通过字段名取数据，不需要从文本里解析。压缩策略在长文本场景下作为兜底手段，能再省 40% 到 50%。两个策略叠加，节点传递的信息量可以控制在原始输出的 10% 到 20% 左右。

四、架构层面的优化

前面两部分（工具定义、对话历史）的优化在单 Agent 方案里也能做。但 Workflow 架构带来了一些"天生如此"的优势：

System Prompt 构成对比

对比项单 Agent ReAct（基线）Workflow Plan AgentWorkflow SubAgent
角色定义~1K（完整人格+灵魂+身份）~0.2K（仅"你是规划 Agent"）~0.2K（仅"你是 xx Agent"）
工具列表~4K（全部工具描述+Schema）无（不调用工具）~1K（只绑当前角色用的 2-3 个）
策略/规则~2K（审批、模式、权限、Skills）~0.8K（拆解规则+颗粒度规则）~0.8K（L1 基础规则+L2 角色规则）
记忆/上下文~2K（记忆检索+历史对话摘要）~0.5K（可用角色清单）~0.5K（当前任务输入）
FewShot无或全部~1K（DAG 反例样本）无
单次调用总计~9-10K（随轮次增长）~2.5-3K（固定）~2.5-3K（固定）

调用轮次的差异

以一个 3 步任务为例：

单 Agent ReAct：

第1轮：System(9K) + 用户需求(0.2K) → 输出 → 上下文变成 10K
第2轮：System(9K) + 上轮历史(10K) + 新结果 → 输出 → 上下文变成 20K
第3轮：System(9K) + 上轮历史(20K) + 新结果 → 输出 → 上下文变成 30K
三轮总消耗：9+10+20+30 = ~69K（历史持续增长）

Workflow：

第1轮：Plan Agent(3K) + 用户需求(0.2K) → 输出 DAG

第2轮：SubAgent A（节点内部也有多步循环）
  第2-1步：System(3K) + 节点描述(0.3K) → 输出 → 上下文变 3.5K
  第2-2步：System(3K) + 上步历史(3.5K) → 输出 → 上下文变 6.5K
  第2-3步：System(3K) + 上步历史(6.5K) → 输出 → 节点完成
  节点内部消耗：3.3 + 6.5 + 9.5 = ~19K

第3轮：SubAgent B（不继承 A 的历史，重新开始）
  第3-1步：System(3K) + 节点描述(0.3K) → 输出 → 上下文变 3.5K
  ...
  节点内部消耗类似，~19K

三轮总消耗：3.2（Plan）+ 19（SubAgent A）+ 19（SubAgent B）= ~41K

差距原因不是"SubAgent 不会累计上下文"——它内部也会循环，也会增长。真正的差距在两点：

  1. System Prompt 更轻（3K vs 9K），每步的基数就小——这主要来自工具定义层的裁剪
  2. 节点间上下文重置——SubAgent B 不会带着 SubAgent A 的历史对话重新开始，每次节点结束历史清零。而 ReAct Agent 的历史会贯穿整个任务，越到后面越重

单次调用： 3K vs 9K，降低约 67%
完整任务（3 步）： 41K vs 69K，降低约 41%

五、未实现的想法

内置命令跳过 LLM

对于流程固定的机械操作（如格式转换、数据校验），目前即使任务很简单，也必须经过 LLM 处理才能执行。一个更好的做法是：在注册工具时加上 skip_llm: true 标记，执行引擎遇到这类任务时直接调用对应的函数处理，不走 LLM。

代码执行替代工具循环

Anthropic 提出过一个思路：不是让 LLM 反复调工具传数据，而是让 LLM 写一段代码去执行多步操作。比如要处理一个大型文档，传统方式是 LLM 调"读取"工具拿到全文，再调"分析"工具传回去——文档内容在两次调用来回中两次经过模型上下文。更好的做法是让 LLM 写一段脚本，在沙箱里一次性完成读取+分析，只把最终结果返回给模型。核心思路：把"LLM 驱动工具"变成"LLM 编写脚本"，减少中间结果反复穿过模型上下文的浪费。

模型分级路由

不是所有任务都需要最强的模型。简单任务（格式校验、简单分类）可以用小模型处理，只有复杂推理（代码审查、报告生成）才调用大模型。目前所有 SubAgent 都用同一个模型，如果按任务复杂度分级路由，在保证效果的前提下还能进一步降低成本。

语义缓存

对于用户重复提出的类似问题，可以用向量检索判断语义相似度，命中后直接返回缓存的回答，不走 LLM。在 RAG 项目的客服场景中，语义缓存的命中率可以达到 35% 左右，能进一步降低 LLM 调用次数。
