一、测什么
方向一：通用 Agent 全维度评测（结果 + 过程）
💡
要点：过程很重要，不要只测结果。
Agent = LLM（感知+规划）+ 记忆 + 工具，是一个复杂决策系统。单纯测结果，难以获得对优化 Agent 的有用信息。

举例：
• 结果正确，但过程错误 — 可能漏做了什么、顺序不对、只是运气好。没有过程数据，这些问题永远发现不了。
• 结果错误，不知道错在哪 — 是没有 plan 好？记忆没获取到？工具调用错了？没有过程记录就无法归因。
结果维度
  • 需求完成度
  • 格式规范
  • 总完成时间
  • 总 Token 消耗
  • 用户满意度
过程维度 （核心）
  • 工具调用
  • 规划链路
  • 记忆能力
  • 异常处理
  • 成本、耗时、安全、权限
方向二：针对性的 Agent 测评
Agent 分为很多类型，不同类型的 Agent 侧重点不同，需要针对性设计测评指标。以下只是几个常见类型的例子，实际项目中要根据业务场景灵活调整。
💻 代码 Agent
  • 代码生成质量
  • 任务完成度
  • 走对工程路径
  • 运行测试验证
💬 客服 Agent
  • 会话解决率
  • 多轮交互质量
  • 记忆召回精度
  • 情绪识别与应对
🔧 运维 Agent
  • 工具调用准确率
  • 异常恢复能力
  • 安全边界遵守
二、怎么测
构建测评数据集
质量比量重要，测评数据集的质量直接决定了评测结果的可信度。
常规样本

常见的用户请求，覆盖核心业务流程。如"查一下我的订单状态"、"帮我重置密码"

边界样本

参数缺失、边界条件等需要追问或特殊处理的场景。如用户只说"帮我查一下上个月那笔退款"但没有提供订单号

对抗样本

恶意输入、越权请求、诱导 Agent 忽略系统规则等安全测试场景

一个好的 Case 有哪些标准？
用户输入：用户会怎么真实表达需求
初始环境：数据库、文件、工具返回值、页面状态
成功标准：什么叫完成任务
失败标准：什么情况必须判失败
评分方式：确定性检查、LLM 裁判，还是人工复核
风险点：是否涉及隐私、权限、资金、删除、外部发送
设计评分器
确定性评分器
  • 是否调用了禁止工具
  • 是否缺少权限校验
  • 是否传了必填参数
  • 是否输出了敏感字段
  • 是否超过最大工具调用次数
  • 是否超过成本阈值
轨迹评分
  • 环（是否有循环调用）
  • 顺序（调用顺序是否合理）
LLM-as-Judge
  • 回答是否真正解决问题
  • 是否遗漏关键解释
  • 是否把不确定结论说得过满
  • 是否用用户能理解的方式解释
  • 是否把工具结果转述错了
选择测评方法
基准测试（Benchmark）

在标准数据集上跑分，与基线版本或竞品横向对比。适合评估 Agent 的通用能力水平，了解当前版本在行业中的位置。

回归测试（Regression）

每次迭代后自动运行，确保新改动不破坏已有能力。Agent 的行为会被 Prompt、上下文、工具和模型共同影响，不做回归测试很难知道哪次改动埋了坑。

AB 测试

线上真实流量分流，对比不同策略/配置的业务效果。能反映 Agent 在真实环境中的表现，但周期较长，适合验证重大改动。

Trace Diff（轨迹对比）

对比新旧版本在同一样本上的执行轨迹差异，直接定位退化原因，而不是猜测"模型是不是变笨了"。

获得任务执行的 Trace
三、一个新人可落地的 Agent 测评流程
1
明确业务目标 — 先写清楚 Agent 到底要帮用户完成什么任务，不要从模型能力出发。
2
收集 20 个真实 Case — 从你自己测试、用户反馈、面试演示场景里整理，不求多，但要有代表性。
3
给每个 Case 写成功标准 — 不要写"回答好"，要写"订单状态变更""测试通过""引用来源不少于 3 个"。
4
设计评分器 — 确定哪些用代码检查，哪些用 LLM 裁判，哪些要人工抽查。
5
保存完整 Trace — 记录每次模型输入输出、工具调用、参数、耗时、Token、错误。
6
每次改动后跑一遍 — 改 Prompt、改工具、换模型、改工作流后，都重新跑黄金 Case。
7
对失败样本做分类 — 是意图识别错、工具选错、参数错、上下文丢失、模型幻觉，还是评分器写错？
8
把线上问题补回评测集 — 用户真实遇到的问题，修完之后要变成新的回归 Case。
四、回归项目
回到我们的项目：Workflow 可视化任务编排与执行系统
测什么
测试的主要方向放在了对比优化上：
Plan 能力：任务分解、DAG 图构建（无环）、冗余步骤、多步连贯性、并发性
任务执行确定化（稳定性指数）：重试次数
成本、效率：总时间、Token 消耗
无效 ReAct 循环占比下降
故障场景流程完整执行率
怎么优化
问题
任务逻辑错误：遗漏步骤、颠倒步骤、无效步骤
DAG 逻辑错误：环路、孤立节点、汇聚分支混乱
输出格式不正确
任务颗粒度失衡
并行度不够
优化方案
Prompt 层
  • 强制输出模板：JsonSchema
  • 拆解规则：检索/工具操作必须在生成类节点之前；独立无依赖任务拆为并行节点
  • 生成规则：不存在循环依赖；所有边指向已定义节点；用户需求全部覆盖，无遗漏关键动作
  • 颗粒度规则：单次工具调用 = 1 个节点；单一独立目标 = 独立节点；禁止跨工具、跨目标合并节点
后置校验层
  • 环路检测
  • 引用合法性检测
  • 步骤完整性检测
FewShot 样本
  • 标准样本：基础串行流程样本、多并行分支样本、带汇聚节点复杂拓扑
  • 反例样本：展示错误 DAG（易出错反向依赖、环路反例）+ 修正后的正确 DAG，让 Plan 规避常见误区
分级 Plan（解决长任务失效问题）
  • 针对长任务，先进行粗任务划分，再调用子 Plan 单独生成 DAG
  • 顶层 DAG 串联各个子流程，避免单次生成超大图导致输出截断、逻辑混乱
双 Agent 机制
  • 引入评审 Agent：独立评审流程逻辑、依赖、环路、需求匹配度，输出修改意见
动态需求兼容
  • Plan 预生成多分支备选 DAG，引擎执行中根据节点输出选择对应分支
  • 三级异常策略兜底
迭代数据闭环优化（长期持续优化）
  • 采集所有校验失败、执行失败的错误 DAG、用户原始需求
  • 自动归类错误类型：环路、步骤遗漏、顺序颠倒、JSON 格式错误
  • 将高频错误案例自动加入 FewShot 样本库，持续迭代 Plan 能力
  • 基于 LLM-as-Judge 量化 DAG 匹配得分，低分样本自动回流训练提示词
