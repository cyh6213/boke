读 Claude Code 源码：我学到的五个设计

2026 年 7 月 1 日 · Claude Code Agent 源码分析

Claude Code 是目前最成熟的 AI Coding Agent 之一。我不满足于"用"，想搞清楚它到底是怎么设计的——为什么它能在长对话中保持稳定、能准确调用工具、能记住用户的偏好。

读完后我收获最大的五个设计点，记录在这里。有些直接借鉴到了我自己的项目里，有些让我重新思考了之前的设计取舍。

一、并行执行：不做 DAG，做连续块

我自己的 Workflow 项目用 KAHN 算法做 DAG 拓扑排序来实现并行调度，每个节点执行后更新下游入度，调度粒度很细。Claude Code 选择了完全不同的思路——连续块（Continuous Chunk）。

它的做法是：按工具调用的原始顺序遍历，判断每个调用是否安全（有没有依赖冲突）。如果安全就并行执行，不安全就等前面的执行完再继续，之后继续找可并行的下一个。

这个方案比我的 DAG 简单太多了。仔细想也确实有道理——在实际场景中，工具调用的依赖关系大多是线性的（读文件 → 改文件 → 执行命令），很少有复杂的多级汇聚或网状拓扑。连续块分区已经能覆盖绝大部分并发场景，实现成本却低很多。

我的思考： 我之前在 Workflow 里设计完整的 DAG 调度器，是一种"为 10% 的复杂场景做了 90% 的工作"。Claude Code 的选择提醒了我：不要高估实际场景的复杂度，先跑起来再优化。如果我要重做 Workflow 的调度层，我会先用连续块方案，然后只在需要的地方引入 DAG。

另一个启发是并发写文件冲突的处理。Claude Code 对多 Agent 并行修改同一个文件采用了加锁策略，这也是最直接的方式。我了解到还有几种思路：

  • 乐观读、悲观写——读不加锁，写时检查版本号
  • 任务分片——调度层面给每个 Agent 分配不同的文件范围，从源头上避免冲突，但可行性不高
  • Git 合并——每个 Agent 在独立分支工作，最后合并。比较成熟但需要用户参与

二、子 Agent：解决上下文污染的最优雅方式

Claude Code 的子代理（Sub-agent）是我觉得设计得最漂亮的一个点。

长对话的核心问题是上下文污染——主 Agent 和工具来回交互产生的消息越来越多，关键信息被淹没，而且工具返回的大量数据会稀释系统指令的权重。

Claude Code 的解法是：给子任务开一个独立的消息列表，跑自己的 Agent 循环，结束后只回摘要给主 Agent。这样主 Agent 的上下文不会被子任务的中间过程污染。

有两个关键的安全设计值得注意：

  • 子 Agent 没有 task 工具，防止无限递归
  • 子 Agent 的工具调用权限更小，不会执行危险操作
和我的对比： 我的 Workflow 项目也有 Sub Agent 机制，但我把 Sub Agent 看作"执行节点的人"，Claude Code 把 Sub Agent 看作"隔离上下文的容器"。这个视角差异让我意识到，Sub Agent 的真正价值不只是任务分工，更是上下文边界管理。

三、内存压缩策略：四层递进，不到万不得已不调用 LLM

这是 Claude Code 能维持长对话稳定性的核心。它的压缩策略分四层，前三层是纯文本操作，不需要花钱调用 LLM：

  • L1 — 裁中间：把旧的对话历史直接裁剪掉
  • L2 — 旧结果占位：大工具输出只保留前 2000 字预览，后面用一行提示代替
  • L3 — 大结果落盘：把大的输出结果先存到磁盘上，防止被后续步骤覆盖

前三层跑完如果上下文还是太大，才触发：

  • L4 — LLM 摘要：花一次 API 调用让模型生成关键信息的摘要，替换所有旧消息

如果 API 还报错，还有应急方案——更激进地从尾部回退裁剪。

这个设计给我的最大启发是：能用文本操作解决的问题，不要用 LLM。LLM 调用又贵又慢，而且摘要本身也要消耗 Token。前三层的文本裁剪和落盘几乎零成本，只有在万不得已时才走 L4。

草稿本（Draft）——让"思考"和"输出"分开

在 L4 摘要生成前，Claude Code 引入了一个我特别喜欢的细节——草稿本机制。

做法是：让模型在输出正式摘要之前，先复盘整个对话，输出一段 <analysis> 分析，然后再输出 <summary>。这个分析在后续处理时会被直接删掉，只保留摘要。

本质上就是让模型"先想清楚再写"。用一次额外思考来换取更好的压缩质量，成本几乎可以忽略（分析内容不保留），但摘要质量明显更高。

九段模板——把摘要结构固定下来

摘要不是自由格式的，而是被固定为九个段落，分三组：

  • 要干什么：目标、背景（讨论过的技术概念）、所有用户消息、待办
  • 干过什么：代码（逐一列举修改的文件和代码段+完整片段）、错误（遇到的所有错误和修复方式，特别是用户让你换方式做的）、已解决的问题
  • 还要干什么：当前状态（压缩发生时正在干什么）、下一步（必须附上最近对话的逐字原文引用）

这个模板解决了摘要最怕的问题——丢失关键信息。固定的结构让模型知道哪些信息必须保留，不会因为"觉得不重要"就跳过。而且"下一步"要求附上原文引用，确保了恢复后能无缝衔接。

我的实践： 我在 MyClaw 的记忆压缩模块里借鉴了这个九段模板的思路，效果很好。固定结构的摘要比自由格式的摘要保留了更多可操作信息，恢复时也不容易"断片"。

四、长期记忆：文件系统即数据库，MEMORY.md 做目录

Claude Code 的长期记忆设计格外实用。核心是一个 MEMORY.md 文件，充当目录角色。每次对话启动时，默认加载前 200 行到 system prompt，让模型知道"有哪些记忆可以用"。

四种记忆类型不是用文件夹物理隔离，而是通过文件名里的 type 字段逻辑标记：

  • user — 用户偏好
  • feedback — 规矩（用户纠正过的）
  • project — 项目背景
  • reference — 线索

写入触发机制有两条路径：

  • 被动存储——用户说了"记住""别忘了"等关键词时，模型主动写入
  • 主动提取——对话结束后，后台自动 fork 一个子 Agent，分析整个对话内容，从中识别出可记忆的信息并自动保存，用户不需要显式说"请记住"

注入机制分两层：

  • 第一层：MEMORY.md 的前 200 行在对话启动时注入 system prompt
  • 第二层：每轮对话前，把一个轻量模型把 MEMORY.md 连同当前用户查询发过去，让它选出最多 5 个最相关的记忆文件，注入到当前上下文

还有一个"后台整理"机制，每 24 小时运行一次，模拟人类睡眠时的记忆整合过程——扫描记忆内容，合并相同内容，移除过时内容，确保 MEMORY.md 的精干。

和我的对比： 我的 MyClaw 项目也做长期记忆，但走的是 Hindsight 的向量+图+时间四路召回路线，架构更重。Claude Code 用文件系统 + 轻量检索就实现了够用的长期记忆。这让我重新思考：Agent 的记忆系统需要多"强"？ 如果你的场景是对话辅助（而非精确的事实检索），文件系统方案的成本和维护复杂度远低于向量数据库方案。

五、System Prompt 组装：动静分离

Claude Code 的 system prompt 不是写死的，而是运行时组装的，分为静态和动态两类。

  • 静态段落：身份定义、语气风格、任务处理原则、工具使用说明——每轮对话都需要的核心指令，内容基本稳定
  • 动态段落：记忆注入、会话指导、MCP 服务器指令、Token 预算控制——根据当前状态决定是否加载

组装逻辑是每轮对话前检查上下文真实状态，基于状态决定加载哪些段落，然后拼接成完整 prompt。

这个设计让我意识到，我之前写 system prompt 都是"写一篇完整的文章放进去"，但 Claude Code 的做法是"搭积木"——需要什么就拼什么。这样每个段落的职责更清晰，也更容易做 A/B 测试和迭代。

读 Claude Code 源码最大的收获不是某个具体技术，而是一种工程取舍的思维方式。它让我看到：

  • 简单方案（连续块）可以覆盖绝大多数场景，DAG 级别的调度不是必须的
  • 文本操作（裁剪/落盘）比 LLM 调用（摘要）便宜得多，优先走低成本路径
  • 文件系统 + 轻量检索在对话场景中够用，不要被"向量数据库"这个 buzzword 绑架
  • 固定结构的模板比自由格式可靠得多

这些取舍背后有一个统一的逻辑：在能用工程解决的问题上，不要依赖模型。
