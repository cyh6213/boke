Project 01

企业知识库 Agentic RAG ↗

项目来源 为什么做这个项目
核心优化 全链路优化
1. 召回率优化 — 提升召回率

为了提升召回率，对整条流水线进行了全链路优化：检索阶段采用问题改写 + BGE 双向量（dense + sparse）混合检索；排序阶段采用 RRF 融合策略 + ReRank 重排。经过这些优化，recall_cap@5 从 68.4% 提升至 81.2%。

2. 幻觉优化 — 降低幻觉率

为了降低生成阶段的幻觉率，在生成环节进行了多项优化：Prompt 优化、结构化输出约束、低分拒答、幻觉检测机制，确保生成内容忠实于检索结果。

3. 智能路由 — 简单问题跳过检索

为了提升召回速度并降低 Embedding API 调用成本，通过智能路由对问题分类：简单问题（问候/数学等）通过关键词/正则匹配直接回答，跳过向量检索，提升召回速度并减少 API 调用。

4. 多跳推理 — 弥补向量 RAG 的全局性问题短板

针对复杂问题（如多跳推理查询），将其分解为子查询链，每步基于前序结论动态构建新查询。例如"程愚瀚和鲁迅的关系"→ 先查程愚瀚（电子科大学生）→ 再查电子科大的人才 → 关联到鲁迅。这一策略弥补了向量 RAG 相较于 GraphRAG 在全局性问题、多跳推理等需要逻辑链条的复杂问题上的短板。

其他优化
上传优化
基于 Redis Bitmap 实现并发分片上传与状态追踪，结合 RocketMQ 异步解析解耦，大文件上传响应速度提升 65%。

MCP
实现 MCP Server，支持外部 AI Agent 通过标准化协议接入查询知识库，打破工具间的数据孤岛。

Project 02

可视化任务编排与执行系统 ↗

从问题到方案

使用 Agent 时发现指令明确但 Agent 不执行——要求检索后回答却跳过检索直接生成，让先做安全扫描再分析代码却直接跳过扫描。这不是"执行策略不一致"的问题，而是Agent 会无视你的指令和规划，自己决定怎么做。

我考虑了两种确定化路径：DAG 约束「整个流程必须做什么」（如强制[读取财报 → 分析指标 → 生成报告]，无法跳过分析），Hooks 约束「每一步前后必须做什么」（如 PreHook 检查文件格式、PostHook 检查敏感数据）。DAG 对"Agent 跳过步骤"的问题更直接，因此选择了 DAG 路径。

将 DAG 图构建交给大模型来完成。Plan Agent 根据用户需求输出 DAG 数据（节点列表 + 边关系），不再参与执行；执行引擎 按图自主调度、分发、并行运算。规划与执行彻底解耦，保证可见即所得。

核心能力
双层并行执行引擎

第一层基于 KAHN 拓扑排序实现节点级并行——入度为 0 的节点直接并发执行，执行后更新下游入度，新的可执行节点继续并发。内置环路检测，通过跟踪入度变化识别死锁。第二层针对单节点内的工具调用实现细粒度并行，形成"节点级 + 工具级"双层并行。经 200 项任务测试，并行执行时间缩短 23%。

节点级断点恢复

摒弃 LangGraph 整体状态快照，为每个节点独立持久化输入输出及状态至 MySQL。运行时数据纯走内存传递，仅断点恢复时从快照重建上下文。支持按请求指定首个失败或未完成节点精准恢复，恢复前可手动修正节点变量，实现更低存储开销与更灵活恢复粒度。

Sub Agent 分发执行与 Token 优化

执行引擎按职责将节点分发给三种 Sub Agent（调研/审查/执行），每类只加载自身角色相关的 System Prompt 和工具描述。建立了公共工具池和私有工具池的划分。相比单 Agent ReAct，单次调用 System Prompt 从 9K 降至 3K（↓67%），完整 3 步任务 Token 消耗降低 41%。

三级异常治理与条件分支路由

固定 DAG 无法像 PAE 动态调整计划。异常治理基于模板方法分三层（重试策略/超时控制/异常分流），支撑直接中断、错误码兜底、失败分支三种策略。条件边分支让 Plan Agent 生成多分支 DAG，每条边带 condition 字段，通过级联裁剪未激活路径，零侵入式支持分支选择。

Plan 能力迭代优化

从 Prompt 约束 → 后置校验 → FewShot 样本 → 分级 Plan → 双 Agent 评审 → 数据闭环采集的完整链路，DAG 构建正确率 97%。基于 Reflexion（NeurIPS 2023）的思想实现迭代闭环，节点分开存储天然保留每次执行的完整 Trace，结合 LLM-as-Judge 低分样本自动回流，形成持续的自改进循环。

MyClaw ↗

项目来源 为什么做这个项目

  • 使用 OpenClaw 发现其记忆写入确定性差（从未主动写入），长期记忆实现过于简朴，想实现一个更好的记忆系统

核心亮点 长期记忆技术选型与测评
四种方案对比测评

通过 RecallBench 利用 LongMemEval 测评集，从 Information Extraction、Multi-Session Reasoning、Knowledge Updates、Temporal Reasoning 四个维度，对比纯模型大窗口 / RAG 检索 / Hindsight 三种方案，评估各方案在长程记忆场景下的实际表现。

最终选型：Hindsight
  • 通过四路召回（向量、关键词、图、时间）策略，提高精确检索能力。
  • 通过四种记忆分类（世界、经验、观点、观察）策略，实现事实和经验的区分。
  • 通过 Opinion 网络的置信度演化机制，新证据强化或削弱已有观点的置信度，实现"用户先喜欢 A 后改口 B"这类矛盾信息的动态追踪与自动更新。
  • 通过为每条记忆携带时间元数据（发生时间 + 记录时间），recall 时结合时间范围过滤与时间衰减排序，实现"用户上个月说了什么"这类时序精确查询。
产品设计 让对话更像真人
  • 自动触发与主动话题：结合 Spring @Scheduled 实现 Heartbeat 定时扫描机制，支持定时触发、空闲触发、被动失效三种策略主动开启话题。对话中结合生活线状态与上下文，在话题自然衰减时平滑过渡到新话题，避免生硬的"对了，我还有个事要说"，模拟真人聊天的联想式话题跳跃，完成从被动响应到主动服务的范式升级。
  • 人物活动轨迹模拟：设计五级生活线体系（Life Plan → Year Plan → Month Plan → Week Plan → Day Plan），Worker Agent 每日定时自动推进角色生活状态，让 AI 拥有连贯的"内在生活"——即使不聊天，角色也在真实地活着，对话时自然分享今日见闻。
  • 时间感知与空间锚定：构建角色的时间认知模型，对话中自然体现时间流逝感（"刚在开车"、"等了我二十分钟"、"昨晚看了一部电影"），结合空间位置锚定，让角色行为在时空维度上自洽，避免"上一秒在做饭，下一秒说在开会"的割裂感。
  • 小说人物迁移：从小说文本自动提取人物档案（性格、经历、社交关系、日常活动），将小说中的时间线映射到真实世界时间轴，实现"小说人物活在现实中"的沉浸式对话体验。
其他亮点
  • 权限控制与人工审批：采用策略防护 + 人工审批（HITL）双层保障机制，Agent 自动执行命令前先经风险策略拦截，高危操作触发人工审批流程，兼顾执行效率与系统安全。
  • 双 Agent 协作架构：设计 Worker + Chatter 双 Agent 协作模式，Worker 后台独立推进剧情与生活线，Chatter 负责与用户对话交互，实现记忆、职责、权限三分离。
  • 上下文组装：参考 OpenClaude 和 Claude Code 的动态 system prompt 拼接思路，Chatter 5 层精简只带聊天必需内容，Worker 在此基础上增加工具、策略、审批等 6 层，实现权限隔离与 Token 优化。
  • 微信绑定：集成微信渠道接入，支持通过微信与 AI 对话，打通 IM 与 CLI 双端交互，用户可在微信端接收主动推送消息与定时提醒，扩展 AI 的可触达场景。
