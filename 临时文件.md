一、测什么 
方向一：通用 Agent 全维度评测（结果 + 过程）
要点
要点：过程很重要，不要只测结果。
原因：Agent=LLM（感知+规划）+记忆+工具。是一个复杂决策系统，单纯测结果，难以获得对优化agent的有用信息。
举例：
● 结果正确，但是过程错误：但是可能漏做了什么，顺序不正确？
● 结果错误，错在哪里？是没有plan好，还是相关信息（记忆）没有获取到？
结果
结果维度（相较于比较简单）：
● 需求完成度 
● 格式规范
● 总完成时间
● 总Token消耗
● 用户满意度
● 等
过程
过程维度：
● 工具调用
● 规划链路
● 记忆能力
● 异常处理
● 成本、耗时
● 安全、权限
● 等
方向二：针对性的agent测评
然后因为agent也是分为很多类型的，所以针对不同类型的agent，我们需要针对性的设计测评指标。比如：
● 代码 Agent — 代码生成质量、任务完成度、是否走对工程路径（如先读测试再定位调用链再改代码）、是否运行测试验证
● 客服 Agent — 会话解决率、多轮交互质量、记忆召回精度、情绪识别与应对
● 运维 Agent — 工具调用准确率、异常恢复能力、安全边界遵守情况

二、怎么测
构建测评数据集
要点：质量比量重要，测评数据集的质量直接决定了评测结果的可信度。
样本类型：
● 常规样本 — 常见的用户请求，覆盖核心业务流程。如"查一下我的订单状态"、"帮我重置密码"
● 边界样本 — 参数缺失、边界条件等需要追问或特殊处理的场景。如用户只说"帮我查一下上个月那笔退款"但没有提供订单号
● 对抗样本 — 恶意输入、越权请求、诱导 Agent 忽略系统规则等安全测试场景
一个case应该包含哪些内容？一个好的case有哪些标准？
● 用户输入：用户会怎么真实表达需求。
● 初始环境：数据库、文件、工具返回值、页面状态。
● 成功标准：什么叫完成任务。
● 失败标准：什么情况必须判失败。
● 评分方式：确定性检查、LLM 裁判，还是人工复核。
● 风险点：是否涉及隐私、权限、资金、删除、外部发送。
设计评分器
确定性评分器：
● 是否调用了禁止工具；
● 是否缺少权限校验；
● 是否传了必填参数；
● 是否输出了敏感字段；
● 是否超过最大工具调用次数；
● 是否超过成本阈值。
轨迹评分：
● 环
● 顺序
LLM-as-judge：
● 回答是否真正解决问题；
● 是否遗漏关键解释；
● 是否把不确定结论说得过满；
● 是否用用户能理解的方式解释；
● 是否把工具结果转述错了
选择测评方法
基准测试（Benchmark）： 在标准数据集上跑分，与基线版本或竞品横向对比。适合评估 Agent 的通用能力水平，了解当前版本在行业中的位置。
回归测试（Regression Testing）： 每次迭代后自动运行，确保新改动不破坏已有能力。Agent 的行为会被 Prompt、上下文、工具和模型共同影响，不做回归测试很难知道哪次改动埋了坑。
AB 测试（A/B Testing）： 线上真实流量分流，对比不同策略/配置的业务效果。AB 测试能反映 Agent 在真实环境中的表现，但周期较长，适合验证重大改动。
Trace Diff（轨迹对比）： 比单次分数更有用的方法是做 Trace Diff——对比新旧版本在同一样本上的执行轨迹差异。通过对比意图识别、权限校验、工具调用、参数传递、最终回答、Token 消耗、耗时等维度的变化，开发者可以直接定位退化原因，而不是猜测"模型是不是变笨了"
获得任务执行的Trace
我的节点输入输出详细记录了
● 用户输入；
● 系统提示和关键配置；
● 模型调用；
● 工具调用；
● 工具参数；
● 工具返回；
● handoff；
● guardrail；
● 错误和重试；
● 最终输出；
● 成本和耗时。
，还包括plan agent输出的执行路径。
三、一个新人可落地的 Agent 测评流程
1. 明确业务目标先写清楚 Agent 到底要帮用户完成什么任务，不要从模型能力出发。
2. 收集 20 个真实 Case从你自己测试、用户反馈、面试演示场景里整理，不求多，但要有代表性。
3. 给每个 Case 写成功标准不要写"回答好"，要写"订单状态变更""测试通过""引用来源不少于 3 个"。
4. 设计评分器确定哪些用代码检查，哪些用 LLM 裁判，哪些要人工抽查。
5. 保存完整 Trace记录每次模型输入输出、工具调用、参数、耗时、Token、错误。
6. 每次改动后跑一遍改 Prompt、改工具、换模型、改工作流后，都重新跑黄金 Case。
7. 对失败样本做分类是意图识别错、工具选错、参数错、上下文丢失、模型幻觉，还是评分器写错？
8. 把线上问题补回评测集用户真实遇到的问题，修完之后要变成新的回归 Case。
三、回归项目
然后再回到我们的项目：workflow 可视化任务编排与执行系统
测什么
我将测试的主要方向放在了对比优化上，
● 作为一个任务编排系统，首先要测试的是他的plan的能力，包括
  ○ 任务分解
  ○ DAG图构建（没有环）
  ○ 冗余步骤
  ○ 多步连贯性
  ○ 并发性
● 任务执行确定化（稳定性指数）
  ○ 重试次数
● 其次是成本、效率上的测试：
  ○ 总时间
  ○ token消耗
●  无效 ReAct 循环占比下降  
●  故障场景流程完整执行率  
怎么优化  
通过 Trace 分析确定问题所在：  
●  Plan 问题 → 优化 Plan Agent 的 Prompt / 增加约束 / 改进任务分解逻辑  
●  执行问题 → 检查 Sub Agent 的指令清晰度 / 工具描述 / 异常处理机制  
●  成本问题 → 合并节点 / 减少不必要的 ReAct 循环 / 优化提示词长度  
●  稳定性问题 → 增加重试策略 / 超时控制 / 降级方案
