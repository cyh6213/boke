## Workflow 项目测试文档（参考业界实践）

### 参考来源（文中标注）

- **腾讯 TEG 网关测试团队**《AI Agent & Skill 测评方案及落地实践》（2026.06）：三类评委组合、基线管理机制、100 分扣分制

### 测试条件

- **测试环境**：本地单机部署，Spring Boot 应用，MySQL 8.0 存储 DAG 和执行快照
- **大模型**：GLM-4-Flash（评测期间固定版本，排除模型升级带来的变量）
- **执行方式**：单元测试模式，外部工具调用使用 Mock 数据
- **稳定性评估**：每个 case 跑 3 次（pass@3），取通过率
- **基线管理**：每个 case 首次执行通过后，经人工确认，该次执行的 Trace 存为基线，后续版本与基线对比做回归验证
- **对比基线**：同一需求用串行方式执行（逐个顺序调工具），对比时间和 Token
- **判定方式**：确定性评分（代码） + LLM-as-Judge（大模型打分）+ 人工抽查 20%

### 一、测试案例设计

#### 1.1 设计原则

参考业界实践，测试集按四个层次构建：

| 层次 | 测什么 | 案例数 | 说明 |
|------|--------|--------|------|
| **触发层** | 该不该触发？ | 8 | 正向（应该触发）和负向（不该触发）用例 |
| **核心逻辑层** | 过程对不对？ | 20 | 覆盖所有核心拓扑结构，用例量最大 |
| **产物质量层** | 产物好不好？ | 12 | 检查 DAG 质量、输出完整性 |
| **异常容错层** | 出错扛得住吗？ | 10 | 输入异常、环境故障、边界条件 |

总样本量：**50 个案例**。每个案例跑 **3 次**（pass@3），完整执行共 150 次。

### 二、评分器详细设计

#### 2.1 评分器架构（三层）

参考 Anthropic 和腾讯 TEG 的组合评分器方案：

| 层级 | 评分器 | 判定方式 | 成本 | 适用场景 |
|------|--------|---------|------|---------|
| 第一层 | 确定性评分器 | 代码自动执行 | 免费、毫秒级 | 结构校验、格式检查、硬约束 |
| 第二层 | LLM-as-Judge | 固定版本大模型 + Prompt | 秒级、有 API 费用 | 需求完成度、颗粒度等软指标 |
| 第三层 | 人工抽查 | 领域专家 | 最贵最慢 | 校准 LLM Judge、诊断异常 |

**评分原则（参考 Anthropic）：**
- **偏重结果而非过程**：评估 Agent "生产了什么"，不强制要求特定工具调用顺序，避免惩罚创造性解决方案
- **不采用二元通过/失败**：多步骤任务应反映成功程度的连续体，授予部分分数
- **防止评分器被绕过**：评分器检查的内容应在任务描述中明确指出

#### 2.2 评分维度与权重

参考腾讯 TEG 的 100 分扣分制，80 分达标线：

| 维度 | 分值 | 评分器 | 评分标准 |
|------|------|--------|---------|
| 需求完成度 | 20 | LLM-as-Judge | 全部功能点正确实现 → 20；部分缺失 → 10-15；完全不符 → 0 |
| 输出格式规范 | 10 | 确定性 | JSON Schema 校验通过 → 10；字段缺失每项扣 2 分 → 保底 0 |
| 结果正确性 | 10 | 混合 | 最终执行结果正确 → 10；部分正确 → 5；错误 → 0 |
| DAG 结构正确性 | 25 | 确定性 | 无节点遗漏/多余、边方向正确、环路检测等 |
| 任务颗粒度 | 10 | LLM-as-Judge | 节点拆分合理、不跨工具合并 → 10；过粗/过细 → 5；失衡 → 0 |
| 工具调用正确性 | 10 | 确定性 | 工具选择正确、参数完整 → 10；每项错误扣 3 分 |
| 异常处理 | 5 | 混合 | 重试/超时/空结果兜底到位 → 5；部分缺失 → 2；全无 → 0 |
| 负向检查 | 10 | 确定性+LLM | 无违规操作（未授权发送/删除/搜索）→ 10；违规操作 → 0 |

#### 2.3 确定性评分器伪代码

```
def score_dag_structure(expected_dag, actual_dag):
    score = 25
    
    # 节点完整性检查
    missing_nodes = expected.nodes - actual.nodes
    extra_nodes = actual.nodes - expected.nodes
    score -= (len(missing_nodes) + len(extra_nodes) * 0.5) * (25 / len(expected.nodes))
    
    # 边方向检查
    for edge in expected.edges:
        if edge not in actual.edges:
            if (edge.to, edge.from) in actual.edges:
                score -= 2  # 反向边
            else:
                score -= 3  # 缺失边
    
    # 结构约束
    if has_cycle(actual.dag): score -= 15
    if has_isolated_node(actual.dag): score -= 10
    if has_invalid_reference(actual.dag): score -= 10
    
    return max(0, score)


def score_tool_correctness(expected_tools, actual_tools):
    score = 10
    for expected, actual in zip(expected_tools, actual_tools):
        if expected.name != actual.name:
            score -= 5
        for param in expected.required_params:
            if param not in actual.params:
                score -= 3
    return max(0, score)


def score_negative_check(actual_dag, user_request):
    score = 10
    # 检查是否包含未授权的操作
    if "send_email" in actual_dag.tools and "发送" not in user_request:
        score -= 10  # 用户没要求发送，Agent 却调用了发送
    if "web_search" in actual_dag.tools and "搜索" not in user_request and "查" not in user_request:
        score -= 5  # 用户没要求搜索，Agent 主动搜索
    return max(0, score)
```

#### 2.4 LLM-as-Judge Prompt

```
你是一个 DAG 质量评审专家。请根据用户需求和生成的 DAG，评估以下维度。

## 维度一：需求完成度（0-20 分）
用户需求：{user_request}
生成的 DAG 节点列表：{node_list}

评分标准：
- 20：全部功能点在 DAG 中都有对应节点，节点顺序合理
- 15：大部分功能点覆盖，少量节点顺序不合理
- 10：部分功能点缺失
- 0：完全不符合需求

> 关注结果而非过程：只要功能点被覆盖，不要求节点命名或顺序与预期完全一致。

## 维度二：任务颗粒度（0-10 分）
每个节点的 description：{node_descriptions}

评分标准：
- 10：每个节点职责单一明确，不跨工具、不跨目标合并
- 5：部分节点过粗（一个节点做多件事）或过细
- 0：颗粒度严重失衡

请输出 JSON 格式：
{"完成度":{"score":N,"reason":"..."},"颗粒度":{"score":N,"reason":"..."}}
```

### 三、稳定性评估（pass@k）

参考 Anthropic 的 pass@k 方法论。每个 case 执行 3 次（pass@3）：

| 指标 | 含义 | 计算公式 |
|------|------|---------|
| pass@1 | 1 次就成功的概率 | 一次通过的 case 数 / 总 case 数 |
| pass@3 | 3 次中至少成功 1 次的概率 | 至少一次通过的 case 数 / 总 case 数 |
| pass^3 | 3 次全部成功的概率 | 三次都通过的 case 数 / 总 case 数 |

**达标线：**
- pass@1 ≥ 80%（基础可用性）
- pass@3 ≥ 90%（高可靠性）
- pass^3 ≥ 70%（生产级稳定性）

### 四、基线管理机制

参考腾讯 TEG 的基线管理做法，建立如下流程：

```
Step 1：初始版本发布后，在所有 case 上执行一次测试
Step 2：人工审核执行结果，确认每一步的 Trace 是正确的
Step 3：将该次执行的完整 Trace 存为"基线"（baseline）
Step 4：后续每次修改（改 Prompt、改代码、换模型）后重新运行
Step 5：对比当前执行与基线是否存在偏差
Step 6：如果出现偏差，判断是回归退化还是合理的改进
Step 7：确认改进后，更新基线
```

基线包含的内容：
- 每个 case 的 DAG 结构快照
- 工具调用序列
- 期望的最终输出
- 耗时和 Token 消耗上限
