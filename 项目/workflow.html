Project 02

可视化任务编排与执行系统

前身 · Workflow 1.0

Workflow 1.0 是一个 AI 工作流编排平台，让用户通过"拖拽"的方式把多个 AI 能力和插件串起来，自动完成复杂任务。

例如：博主构建播客工作流 — 发送文章 → 大模型处理 → 语音插件转音频 → 发送博客，全部由用户可视化拖拽定义。
核心亮点：业务人员无需写代码，仅需在界面上编排节点即可搭建 AI 应用。
局限性：不智能，所谓的AI能力仅仅是LLM最基本的文字处理，无 ReAct、无 PAE。所有流程和参数需用户手动定义。
涉及功能：DAG 图构建 · 节点 JSON 格式 · 节点状态 · 串行执行 · 大模型接入 · 插件接入 · 异常处理 · SSE 流式输出
项目来源 从问题到方案
要解决的问题：Agent 不按照需求执行

使用 Agent 时发现指令明确但 Agent 不执行。比如要求"检索后回答"结果跳过检索直接生成答案。这不是"同一个任务每次执行策略不一致"的问题——真正的问题是：Agent 会无视你的指令和规划，自己决定怎么做。

❌ 没有 DAG："检索竞品资料后生成分析报告" — Agent 直接生成了一份报告，根本没去检索，因为它觉得"自己知道竞品信息"。
❌ 没有 DAG："先查天气再安排行程" — Agent 查完天气后直接给出行程，忘了查交通，因为中间没有节点强制它做这件事。
❌ 没有 DAG："分析代码质量前先做安全扫描" — Agent 跳过安全扫描直接分析，因为它的内置判断认为"这个项目看起来没什么安全问题"。
✅ 有 DAG 后：Plan Agent 根据你的需求动态生成执行计划，执行引擎强制按计划执行，Agent 在每个节点只能做该节点的事，不存在"自己决定跳过某一步"的可能。
两种确定化路径

针对 Agent 执行不可控的问题，我考虑了两种不同的确定化路径：

DAG — 约束「流程」

限制 Agent 在整个流程中必须要做什么。

举例："帮我分析财报并写报告" — DAG 强制流程为 [读取财报 → 分析指标 → 生成报告]，Agent 不能跳过「分析」直接写。
✅ 保证"该做的事都做了"
Hooks — 约束「操作」

限制 Agent 在某一步前后必须要做什么。

举例：执行「分析指标」之前 — PreHook 检查文件格式；之后 — PostHook 检查输出是否含敏感数据。
✅ 保证"每一步都做得规范"
我选择了 DAG 路径，因为流程层面的约束对"Agent 不按需求执行"这个问题更直接。同时 Hooks 的思路可以后续补充，两者不冲突，可以结合。
我的解决方案
💡 有了 Workflow 1.0 的铺垫，自然而然地想到了将 DAG 图构建的步骤交给大模型来完成，与 PAE 架构联系起来。

摒弃 Master 集中调度。Plan Agent 只负责根据用户任务逻辑，输出一份完整、结构化的 DAG 数据（节点列表 + 边关系 JSON），不再参与后续调度；执行引擎 读取这份独立 DAG 数据，自主完成全流程调度、节点分发、并行运算。总结一下就是把"人去定义执行计划"的过程交给了 Plan 节点。

好处
将 Agent 执行任务的流程固定化，DAG 图可视化后可以清楚判断任务执行逻辑是否是我们想要的，保证可见即所得，不会出现"明面一套背地一套"的情况。
坏处
相比 PAE 框架可根据前一步结果动态改变下一步计划，本方案是固定化的。但针对节点执行失败的情况，支持三种异常处理机制确保流程完整执行。
双层并行 双层并行执行引擎

基于 KAHN 拓扑排序实现 DAG 节点的批次拆分与并行调度，入度为 0 的节点自动并发执行。同时针对单节点的工具调用层面，实现细粒度并行，形成"节点级 + 工具级"双层并行机制。经 200 项任务评估集测试，相比串行基线，并行执行时间缩短 23%。

第一层 · 节点级并行 — KAHN 拓扑排序
Kahn 算法，核心思路是基于入度的拓扑排序。入度（In-Degree）指一个节点被多少条边指向，入度为 0 表示该节点没有任何前置依赖，可以直接执行。
并行执行逻辑：入度为 0 的节点可以直接执行，每执行一个节点就将其指向的节点的入度减一，于是就会有新的可执行节点出现。将节点分为一批一批执行，实现更大程度的并行。
环路检测：记录每个节点的入度，不断移除入度为 0 的节点并更新下游入度。如果执行完毕后仍有节点入度不为 0，说明存在环路。
第二层 · 工具级并行
除了节点任务层级的并行外，还有颗粒度为单个工具调用层面的并行，比如同时读多个文件。两者结合形成双重的并行策略。
并行输出隔离

多线程并行场景下如果直接往 System.out 打印，由于线程调度抢占，不同任务的日志会交错混杂，可读性很差。

我们的做法是做输出隔离，给每个执行步骤单独分配独立的 ByteArrayOutputStream 内存缓冲区，每个任务执行时不直接写标准输出，全部写入自己专属的本地缓冲，从根源上避免多线程输出互相干扰。

等所有并行任务全部执行完毕后，再按照业务定义的步骤原始顺序，依次把每个缓冲区的内容批量刷新到标准输出。这样既保留了多线程并行执行的性能优势，又能保证最终日志按固定顺序完整输出，每条任务的日志整块连贯，不会出现穿插乱序的情况。

断点恢复 节点级断点恢复

摒弃 LangGraph 整体状态快照，为每个节点独立持久化输入输出及状态至 MySQL。运行时数据走内存传递，仅断点恢复时从快照重建上下文，可按请求指定首个失败或未完成节点精准恢复，同时支持恢复前手动修正节点变量。

节点级回滚与断点恢复
每个节点的输入、输出、状态单独存储

三张表各司其职：workflow.flow_data（JSON 列）存 DAG 拓扑结构，execution_snapshot 存每个节点的输入输出和状态，execution_record 存整次执行的全局汇总。

运行时的数据传递纯走内存 HashMap，节点执行完顺手把快照同步写入 MySQL，但不从数据库读。

只有断点继续触发时，才从 execution_snapshot 读出快照恢复到内存，结合 flow_data 重新解析的拓扑图确定从哪个失败节点开始，跳过已成功的，接着往下跑。最后把 execution_record 状态从 FAILED 改回 RUNNING，执行完再更新。

断点恢复流程
1
查询执行记录，校验归属
2
重新解析 DAG，进行拓扑排序
3
加载该执行记录的所有执行快照
4
确定从哪个节点恢复开始
优先级：请求指定 > 第一个失败的节点 > 第一个未完成的节点
5
构建恢复状态
收集恢复起始节点之前所有成功节点的输出
恢复条件节点的分支选中信息（跳过的节点）
6
保存用户修改的变量
7
更新执行记录状态为"运行中"
8
从恢复状态开始执行
LangGraph 的断点继续实现方式

LangGraph 的断点继续基于内置的 checkpoint 机制，核心是"整体状态快照"而非"逐节点记录"。

框架内部维护一个全局 AgentState，所有节点共享同一个状态字典。每当一个或一批并行的节点执行完毕，就自动生成一个 checkpoint。恢复时定位到最近的 checkpoint，接着往下跑。

之所以用整体快照而不是逐节点存，是因为 LangGraph 的节点间没有"我这个节点的输出属于我"的概念。所有节点的输出都写进同一份全局状态的不同字段。

对比
DAG 图方式更优
存储开销 — 只需记录每个节点的输入输出和状态，存储效率高。LangGraph 每次保存完整全局状态快照，存储开销大。
恢复灵活性 — 可精确指定从任意节点开始，恢复前可手动修改节点输出或变量。LangGraph 只能从最近 checkpoint 继续，无法跳过或修改中间节点。
调试粒度 — 清晰看到每个节点的执行历史和输入输出，排查问题直观。LangGraph 只能看到完整状态，单节点调试不如 DAG 图方便。
LangGraph 更优
实现复杂度 — DAG 图需设计多张表、处理节点依赖、收集恢复状态，实现复杂。LangGraph 框架自动维护状态和 checkpoint，实现简单。
状态一致性 — DAG 图手动构建恢复状态时易出错，特别是复杂依赖场景。LangGraph 每次保存完整快照，恢复时直接加载，不易出错。
动态结构支持 — DAG 图拓扑预定义，动态分支和循环处理困难。LangGraph 整体快照天然支持循环、动态分支等复杂结构。
各有利弊
适用场景 — DAG 图适合流程固定、需要精细控制恢复点、存储资源有限的场景；LangGraph 适合复杂动态工作流、追求开发效率、需要时间旅行调试的场景。
Sub Agent Sub Agent 分发执行与 Token 优化
Sub Agent 分类与职责

Plan Agent 生成 DAG 后，执行引擎按节点职责将任务分发给不同角色的 Sub Agent。目前系统设计了三种 Sub Agent，各司其职：

调研 Agent — 负责搜索与信息收集，绑定搜索类工具（WebSearch、文件读取）。不参与执行和决策。
审查 Agent — 负责代码审查与安全扫描，绑定代码检查类工具、安全漏洞扫描规则。不参与代码生成。
执行 Agent — 负责执行命令、写文件、生成报告等实际操作，绑定执行命令和文件写入类工具。不参与规划和审查。
Token 消耗对比

Sub Agent 只加载自己角色需要的 System Prompt 和工具描述，不加载全局规则和其他角色的工具。跟一个 Agent 用 ReAct 执行所有任务的方案对比：

单 Agent ReAct
System Prompt: ~9K（含全部工具、策略、记忆）
历史不断累积，轮次越多越重
Sub Agent 方案
System Prompt: ~3K（只含当前角色规则和工具）
节点结束后重置，历史不跨节点累积

单次调用：3K vs 9K，降低约 67%
完整任务（3 步）：41K vs 69K（Sub Agent 内部也有循环，但节点间上下文重置阻止了历史持续膨胀），降低约 41%
详细数据可参考文章《Agent Token 优化实践》。

异常治理 三级异常治理与条件分支路由

本方案采用固定 DAG 执行，相比 PAE 架构可根据前一步结果动态修改下一步计划的方式，丧失了执行中调整流程的能力。三级异常治理和条件分支路由就是为了弥补这一限制而设计的——异常治理处理"节点执行失败怎么办"，条件分支处理"执行结果不同如何选择路径"。

三级异常治理

基于模板方法抽象通用执行基类，自上而下封装重试策略、超时控制与异常处理。

顶层 · execute（重试）

总入口。需要重试时通过循环调用 doExecuteWithTimeout，无需重试则直接执行。重试间隔支持固定、线性、指数退避三种策略。

中层 · doExecuteWithTimeout（超时）

通过 AsyncUtil 工具类将节点执行放入限时任务中，到达设定时间直接抛出超时异常。

底层 · doExecute（异常处理）

执行节点逻辑，捕获所有异常后统一包装，按配置分三种处理方式：

① 直接中断 — 关键节点失败后整体进入失败状态。适用于支付、扣款、核心写操作等。
② 错误码策略 — 按预配置的错误码或默认值继续执行。适用于辅助查询、非核心推荐等。
③ 失败条件分支策略 — 引导到异常处理分支，用于告警、补偿、回滚或特定兜底逻辑。
条件分支路由

Plan Agent 可生成多个分支备选 DAG，执行时根据节点输出选择对应分支。实现方式是在每条边上加 condition 字段——节点执行完成后，只有匹配条件的边被"激活"，仅减少激活路径上节点的入度。

未选中分支的节点永远不会入度为 0，自然不会被调度执行。通过这种 级联裁剪 策略，可以在不修改 KAHN 调度器的情况下原生支持分支与选择。

Plan 优化 如何优化 Plan Agent 的 DAG 构建能力
问题分析
任务逻辑错误 — 遗漏步骤、颠倒步骤、无效步骤
DAG 逻辑错误 — 环路、孤立节点、汇聚分支混乱
输出格式不正确
任务颗粒度失衡 — 节点太大或太小，影响并行度和可复用性
并行度不够 — 未能充分利用 DAG 的并发能力
优化手段
Prompt 层
  • 强制输出模板 — JsonSchema 约束输出格式
  • 拆解规则 — 检索/工具操作必须在生成类节点之前；独立无依赖任务拆为并行节点
  • 生成规则 — 不存在循环依赖；所有边指向已定义节点；用户需求全部覆盖
  • 颗粒度规则 — 单次工具调用 = 1 个节点；单一独立目标 = 独立节点；禁止跨工具、跨目标合并节点
后置校验层
  • 环路检测
  • 引用合法性检测
  • 步骤完整性检测
FewShot 样本
  • 标准样本 — 基础串行流程、多并行分支、带汇聚节点复杂拓扑
  • 反例样本 — 展示错误 DAG（反向依赖、环路反例）+ 修正后的正确 DAG，让 Plan 规避常见误区
分级 Plan（解决长任务失效问题）
  • 针对长任务先进行粗任务划分，再调用子 Plan 单独生成 DAG
  • 顶层 DAG 串联各个子流程，避免单次生成超大图导致输出截断、逻辑混乱
双 Agent 机制
  • 引入评审 Agent，独立评审流程逻辑、依赖、环路、需求匹配度，输出修改意见
迭代数据闭环优化 — 基于 Reflexion 的自改进机制
参考 Reflexion (NeurIPS 2023)：通过语言反馈替代权重更新的强化学习范式
  • Actor → Evaluator 闭环：采集所有校验失败、执行失败的错误 DAG 和用户原始需求，作为 Evaluator 的负样本输入
  • Self-Reflection 错误归因：自动归类错误类型——环路、步骤遗漏、顺序颠倒、JSON 格式错误，提炼可复现的失败模式
  • Experience Replay 经验回放：将高频错误案例自动加入 FewShot 样本库，等价于 Reflexion 中将反思注入长期记忆，持续迭代 Plan 能力
  • LLM-as-Judge 量化评估：基于 LLM-as-Judge 量化 DAG 匹配得分，低分样本自动回流至 Prompt 优化流程，形成完整的 Verbal Reinforcement Learning 循环
