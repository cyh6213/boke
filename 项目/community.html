开发者技术社区平台

项目来源 为什么做这个项目

面向互联网开发者的内容分享平台，用于帮助开发者成长。项目为前后端分离架构。

Spring Boot MyBatis-Plus MySQL Redis Caffeine RocketMQ Docker
主要职责
  • 使用 RocketMQ 异步处理点赞、评论、收藏、消息推送，实现核心业务与统计、通知模块解耦。
  • 基于 Redis 实现白名单控制、用户活跃度排行、计数统计，提升接口响应与体验。
  • 设计 Caffeine+Redis 多级缓存，解决首页、专栏等热门数据吞吐量瓶颈，同时保证缓存一致性。
  • 基于 AOP + 自定义注解 + MDC + TraceID 实现日志全链路追踪，支持动态业务码（bizCode）注入、方法耗时统计与性能监控；基于 Logback 自定义追加器实现异常邮件告警。
  • 通过 Redis Pipeline 批处理、@Async 异步执行、并行访问优化等手段，降低网络开销，解决高并发下计数统计、消息通知等场景的性能瓶颈，减少 60% 页面初始化时间。
  • 编写 Docker Compose 编排脚本，实现"一键拉起"开发环境，缩短本地环境搭建时间。
记忆 基于 Redis 实现数据控制与统计
白名单 基于 Set 的白名单控制

基于 Redis 的 Set 实现白名单控制。如果在白名单里，文章就直接发布上线；如果不在，就进入审核队列，等人工审核通过再上线。

底层原理
Set 的底层是 dict，查询的时间复杂度是 O(1)，能极快地判断元素是否存在，而且天然去重，不用担心重复添加。
延伸：这种机制也可以反过来用，改成黑名单来实现禁言或封禁。除此之外，Set 还能用于防止点赞或收藏重复操作，找共同好友等。
活跃度排行 基于 ZSet 的活跃度排行

基于 Redis 的 ZSet 实现活跃度排行，支持按照分数进行排序。

积分规则
按行为权重设计：访问页面加 1 分，点赞收藏加 2 分且取消时收回，评论加 3 分，发布审核通过的文章加 10 分。
幂等策略
用 Redis Hash 存储用户当天的行为记录，加分前先检查，不存在才执行。针对原子性问题（幂等判断加更新不是原子操作），用 Lua 脚本把整个逻辑封装起来，确保整条逻辑的原子性。
过期策略
幂等记录设置 1 天过期，日榜设 31 天，月榜设 1 年。不存数据库是因为排行榜读写频率极高，属于运营辅助数据，不是核心业务，对持久性要求不高。
计数统计 基于 Hash 的计数统计

选择用 Redis 而不是数据库，主要是因为数据库在高并发下做 count 或者实时更新字段，性能差且 IO 开销大。而 Redis 基于内存，INCR 这类原子命令天然适合频繁的计数场景，同时也能有效减轻数据库的压力。

为什么用 Hash 结构
对于一个用户或一篇文章，往往有多个统计字段，比如点赞数、评论数、收藏数。用 Hash 可以把它们放在同一个 Key 下，节省 Key 数量，还便于统一管理。
缓存一致性 Caffeine + Redis 多级缓存

项目整体是本地缓存 Caffeine（一级）+ 分布式缓存 Redis（二级）+ 数据库 MySQL 的三级架构。

两级缓存分工
Caffeine（一级） — 存高访问、低变更的数据，比如首页文章列表、专栏教程等。
Redis（二级） — 存需要共享、更新频率高、一致性要求高的数据，比如排行榜、热点文章，计数类操作，配合 Pipeline 做批量优化。
数据一致性策略
采用 先更新数据库再删缓存 + 过期时间 策略：
① 更新数据时先更新数据库
② 删除 Redis 缓存
③ 删除 Caffeine 本地缓存
④ 同时给缓存设置合理过期时间，作为兜底保证最终一致
几种缓存策略对比
✅ 先更新数据库，再删缓存（项目选型）
可能的问题场景：缓存刚好过期 → 请求 A 查库读旧值 → 请求 B 更新数据库并删缓存 → 请求 A 将旧值写回缓存。但概率极低，要求三个条件同时满足：缓存刚好过期、有并发读写、写操作在读操作的"查库"和"写缓存"之间完成。即使出现，下次缓存过期后自动修复。
🔄 异步更新（Canal + binlog）
先写 MySQL，利用 Canal 监听 binlog，异步删除 Redis 缓存。追求最终一致性，好处是业务代码不用关心缓存，解耦，稳定性更高。
🔒 强一致性方案
加锁保证同一时间只有一个人能操作数据，代价是性能下降明显。关键数据直接查库，不走缓存。
❌ 三种有问题的策略
先更新缓存，再更新数据库 — 并发 + 网络延迟导致顺序错乱，且更新完缓存后数据库操作可能失败。
先更新数据库，再更新缓存 — 同样并发 + 网络延迟导致顺序错乱。
先删缓存，再更新数据库 — 在读多写少场景下，删缓存后另一个请求读到旧值并回写，导致脏数据。
日志 全链路日志追踪与异常报警
增强日志（MDC + TraceID + bizCode）

基于 AOP + 自定义注解 + MDC + TraceID 实现日志全链路追踪，支持动态业务码（bizCode）注入、方法耗时统计与性能监控。

执行流程
① 请求进来后，Filter 生成 TraceId 存入 MDC（SLF4J 提供的 ThreadLocal 容器）
② 方法上标注 @MdcDot 注解时，AOP 切面拦截，解析注解里的 bizCode（如文章 ID），也存入 MDC
③ 打印日志时统一输出 TraceId 和 bizCode
④ 请求结束后清理 MDC 防止内存泄漏
TraceId vs bizCode
TraceId — 一次请求的唯一标识，用来串联整个调用链的所有日志。生成规则：机器 ID + 线程 ID + 时间戳 + 自增序列号，保证全局唯一。
bizCode — 业务唯一标识，比如文章 ID 或用户 ID，方便按业务检索日志。
异常报警

基于 Logback 自定义 Appender 实现异常邮件告警。

实现原理
自定义类继承 AppenderBase，重写 append 方法。Logback 每输出一条日志就会调用这个方法。方法内做频率控制：用静态变量 lastSendTime 记录上次发送时间，每次判断当前时间减去上次发送时间是否大于 10 分钟，满足条件才发送告警邮件并更新时间，否则忽略。这样就实现了 10 分钟内只发一次邮件的频率限制。

邮件发送利用 Spring 封装的 JavaMailSender：在 append 方法里从 ILoggingEvent 对象获取日志级别、消息内容、时间、类名等信息，拼接成邮件标题和正文后发送。
性能优化 解决高并发性能瓶颈
批处理 Redis Pipeline 批量操作

Redis Pipeline 把多条 Redis 命令打包在一起，通过一次网络请求发送给服务端，减少网络往返耗时。主要用在文章阅读量累加、点赞收藏、活跃度统计这类计数场景。

注意事项
Pipeline 不具备原子性，不能当成事务来用；一次性打包的命令不能过多，否则会占用服务端缓冲区；同时不支持实时返回执行结果。
异步执行 @Async + Listener

@Async 注解搭配 Listener 使用。NotifyMsgListener 负责监听评论、点赞等业务事件，异步生成通知消息并推送给用户。没有选择 MQ 是因为 Listener 非常轻量简单，用在评论点赞等对一致性要求不高的场景十分合适。

线程池配置
@Async 默认创建新线程，高并发会导致 OOM。因此自定义了一个配置类，专门建了一个线程池（IO 密集），任务队列有限长度，拒绝策略改成不处理，避免占用太多资源。
并行访问 CompletableFuture 并行优化

页面加载慢时，除了加机器和加缓存，把串行改成并行也很有效。将获取页面不同数据封装为 CompletableFuture 任务，使用 allOf 方法组合并发执行，通过 join 阻塞等待。所有数据收集完成后再返回给前端，大约减少了 60% 的页面初始化时间。

监控 Actuator + Prometheus + Grafana

搭建 Actuator + Prometheus + Grafana 监控体系，支持异常告警，并使用 JMeter 完成性能压测与容量评估。

工作流程
Actuator — Spring Boot 提供，负责暴露应用的内存、线程等内部数据。
Prometheus — 定期从 Actuator 拉取指标数据并存储。
Grafana — 可视化，连接 Prometheus 作为数据源，将指标以图表、仪表盘形式展示。
JMeter — 独立压测工具，模拟高并发请求，测试系统在压力下的表现。

先用 JMeter 对应用施压，同时 Actuator 不断暴露运行指标，Prometheus 定时抓取并存储，Grafana 实时展示成仪表盘。通过观察压测过程中的 CPU、内存、QPS、响应时间等指标变化，判断系统瓶颈。
