Project 03

Tree — 长链路智能体底层框架

项目来源 怎么想到做这个 3.0
从招聘需求到项目灵感

之前开发过代码智能体、浏览器操作智能体、计算机操控 Agent、深度调研类长链路智能体。 在开发过程中，我注意到长链路 Agent 普遍会遇到推理漂移、任务遗忘、上下文溢出等问题。

💡 我忽然想到：我的 Workflow 项目（DAG 图）是不是可以改造成一个长链路 Agent 的底层框架？

长链路 Agent 遇到的问题本质上都是"上下文太长 → 关键信息被淹没"。DAG 图天然适合做这件事——按节点独立存储、外层做摘要、内层保留细节，既解决了上下文溢出，又保留了完整轨迹。
问题 长链路智能体需要解决的问题
上下文窗口溢出
长链路执行过程中历史记录超出模型上下文限制
遗忘与衰减
早期的重要信息被后续步骤淹没，Agent 忘记任务初心
推理漂移
Agent 在长链路中逐渐偏离原始任务主线
缺失关键节点
Agent 跳过某些必要步骤，执行不完整
错误累计
早期的小错误在后续步骤中被逐步放大
循环冗余检索
Agent 重复执行相同工具调用，浪费 Token
多源信息冲突
不同来源的检索结果矛盾，Agent 难以判断
异常不可自愈
中途出错后无法恢复，只能重头再来
核心判断：上下文窗口溢出、遗忘衰减、推理漂移属于同一类问题——上下文太长导致关键信息被淹没。这正是 DAG 图可以解决的。
思路 我的思路：多层 DAG 树
🌳 多层 DAG 树
外层 DAG

Agent 能看见的完整任务结构。每个节点是抽象的任务块，包含这一步做了什么、核心产出是什么。 外层摘要保证了"任务初心"始终可见，解决遗忘和漂移。

▼ 点击节点展开 ▼
内层 DAG（子图）

每个外层节点内部是一个完整的子 DAG，记录了这个任务块内部的所有细节操作——读了哪些文件、调用了哪些工具、中间结果是什么。 渐进式输出：先看外层摘要了解全貌，再深入某个节点查看细节。

▼ 再展开 ▼
叶子节点（最小粒度）

最底层的 Agent 执行单元。记录了某一步具体做了什么——比如读取了某个文件、执行了某个搜索查询。 这是可审计的最小单元。

🔴 运行时错误标记（核心创新）

传统方案是任务执行完毕后扫描日志来发现问题。Tree 的做法是在运行时实时标记异常：

执行异常标记 — 节点执行失败时，运行时自动标记并记录上下文
决策分支标记 — Agent 在何处改变了策略、走了哪条分支、为什么
复盘与优化 — 查看哪些节点产生了分支 = 哪里遇到了问题，针对性优化
🔄 动态纠错：固定主干 + 动态分支

一部分图是规划好的、不变的（比如核心搜索步骤）。另一部分（细节实现）是执行时动态生成的。 遇到问题时产生分支，切换到另一条路径继续执行。

决策点可见：Agent 可能在任务执行中为了达成结果而"抄近路"——结果看似正确，但过程不是我们想要的。 通过查看决策分支，可以定位到哪一步换了策略。
优势 核心优势
多层树 → 渐进式输出
外层摘要是大局观，内层是细节。Agent 始终能看到任务全貌（解决漂移），又可以点进去看任意节点的详细执行过程。天然符合"先看整体再看局部"的用户认知。
节点独立存储 → 上下文溢出 + 异常自愈
继承 Workflow 2.0 的按节点存储设计。每个节点只关注自身上下文，不存在"所有历史塞进一个窗口"的问题。出现异常时可以从任意节点回溯重跑。
决策点可见 → 可审计
Agent 在哪一步改变了策略、走了哪条分支、为什么走这条路——全部可见。复盘时通过查看分支就能定位问题点。
运行时标记 → 精准排错
错误在运行时实时标记，而不是事后扫描。结合分支信息，可以精确知道"哪一步执行了什么、产生了什么分支、因为什么失败"。
场景 场景分析：Deep Search
为什么选 Deep Search 作为第一个场景？

Deep Search 对 Agent 自身的执行能力要求相对较低（主要是检索和整理），但对长链路稳定性、方向控制、信息综合要求很高。非常适合作为 Tree 的验证场景。

通用问题 → Deep Search 场景具体表现
通用长链路问题 Deep Search 场景下的具体表现
目标漂移 搜索方向逐渐偏离原始主题
上下文衰减 早期搜索结果被后续步骤淹没
错误累积 一次错误搜索导致后面全错
异常不可恢复 搜索中途失败，重头再来
+ 特有挑战 搜索方向选择 / 深度/广度权衡 / 多源信息冲突 / 终止条件判断 / 环境波动
对比 与现有方案对比

了解现有项目是如何解决长链路 Deep Search 问题的，再思考 Tree 的差异化：

dzhng/deep-research
用 depth/breadth 参数解决深度/广度权衡，用递归解决方向选择。 —— 缺少执行过程的可见性，无法审计"为什么这么搜"
DeerFlow v1
用多角色分工解决上下文衰减（每个角色看自己的上下文），用 Planner 解决目标漂移。 —— 架构复杂，节点间信息传递不透明
ODS（推理模型方案）
用推理模型（DeepSeek-R1）本身的推理能力解决搜索方向选择和终止条件判断。 —— 依赖模型能力，不稳定且成本高
Tree 的差异化定位：
不依赖特定模型能力，通过 DAG 树结构本身来解决长链路问题。 核心思路是"用结构保证稳定性"——把长链路拆成多层摘要 + 细节，让 Agent 在任何步骤都能看到任务全貌。 而且执行过程完全透明、可审计。
