MyClaw — AI 长期记忆系统

项目来源 为什么做这个项目
技术层面
  • 前面已经做了 Workflow 和 RAG，缺一个跟记忆相关的项目，对记忆方向比较感兴趣
  • 使用 OpenClaw 发现其记忆写入确定性差（从未主动写入），长期记忆实现过于简朴，想实现一个更好的记忆系统
产品层面
  • 希望设计一个更像真人朋友的对话系统，让对话更真实、更有依据
核心亮点 长期记忆技术选型与测评
📦 测评数据来源
基于 LongMemEval 基准测试集（ICLR 2025），通过 RecallBench 框架，从以下四个维度对比纯模型大窗口 / RAG 检索 / Hindsight 三种方案在长程记忆场景下的实际表现：

• Information Extraction（精确检索）— 例如："用户上次聊天时提到最喜欢吃什么菜？"
• Multi-Session Reasoning（跨会话推理）— 例如："结合前两次对话，用户最终决定选哪个方案？"
• Knowledge Updates（知识更新）— 例如："用户之前说喜欢 A，后来改口说 B 更好，现在应该推荐哪个？"
• Temporal Reasoning（时序推理）— 例如："用户的职业经历按时间顺序是怎样的？"

🔍 结果分析 — 为什么 Hindsight 全面领先？
● Information Extraction（精确检索）
RAG 检索（65%）在此维度天然占优，因为精确检索本质就是"问题 → 检索 → 匹配"的过程，与 RAG 的核心机制一致。但 RAG 受限于检索精度（chunk 切分、向量匹配误差），仍有遗漏。
Hindsight（95%） 通过四路召回（向量 + 关键词 + 图 + 时间）大幅提升召回覆盖率，配合时间衰减排序精确命中目标，因此远超其他方案。
● Multi-Session Reasoning（跨会话推理）
纯模型大窗口（45%）和 RAG 检索（50%）表现接近且偏低。大窗口模型存在"位置偏见"——更关注近期文本，早期会话信息被稀释；RAG 检索以 chunk 为粒度检索，丢失了会话间的时序关联和逻辑链条，无法串联多轮信息。
Hindsight（90%） 通过四种记忆分类（世界/经验/观点/观察）将跨会话信息结构化存储，结合置信度演化机制追踪观点变化，能有效串联多轮对话中的推理链路。
● Knowledge Updates（知识更新）
这是所有方案差距最大的维度。RAG 检索（45%）甚至低于纯模型大窗口（50%），原因在于 RAG 无法判断信息优先级——新旧文档同时被召回后缺乏冲突解决机制，旧信息反而干扰了判断。大窗口模型同样面临"位置偏见"，虽更关注近期信息，但无法理解"用户从喜欢 A 变为喜欢 B"这一更新操作本身。
Hindsight（88%） 的 Opinion 置信度演化机制 专为此场景设计：新证据补充或否定旧观点时，动态调整置信度分数，旧观点被削弱而非直接删除，保留完整的认知演变轨迹。
● Temporal Reasoning（时序推理）
RAG 检索（50%）最弱——检索到的是碎片化文档，无法还原完整时间线。大窗口模型（55%）稍好，因为长上下文可以容纳时间线信息，但多跳串联（"A 事件 → B 事件 → C 事件"的因果链）仍然困难。
Hindsight（90%） 为每条记忆携带时间元数据（发生时间 + 记录时间），recall 时结合时间范围过滤与时间衰减排序，天然支持按时间轴检索和推理。
最终选型：Hindsight
  • 通过四路召回（向量、关键词、图、时间）策略，提高精确检索能力。
  • 通过四种记忆分类（世界、经验、观点、观察）策略，实现事实和经验的区分。
  • 通过 Opinion 网络的置信度演化机制，新证据强化或削弱已有观点的置信度，实现"用户先喜欢 A 后改口 B"这类矛盾信息的动态追踪与自动更新。
  • 通过为每条记忆携带时间元数据（发生时间 + 记录时间），recall 时结合时间范围过滤与时间衰减排序，实现"用户上个月说了什么"这类时序精确查询。

短期记忆

短期记忆的实现很简单，就是一个滑动窗口。没有做压缩。

原因：闲聊场景下，太久的对话内容已经没有保留在 Prompt 里的价值。而之前的事情是靠 Hindsight 从对话中提取并长期存储的事实，而不是把昨天的对话原文塞回来。
上下文组装

1. 设计思路 — 参考了 OpenClaude 和 Claude Code 中关于系统提示词动态构造的实践，实现 system prompt 的动态拼接。

2. 双 Agent 差异 — 两个 Agent 职责不同，组装结果也不同。Chatter 精简、Worker 完整，具体对比如下：

为什么拆双 Agent：使用 OpenClaw 时发现，一边想让 AI 执行任务，一边想让它作为朋友自然对话，但同一个 Agent 执行任务后对话人格会变得奇怪。因此设计了 Worker + Chatter 协作模式：Worker 后台独立推进剧情与生活线，Chatter 负责用户对话交互，实现记忆、职责、权限三分离。
好处：
• Chatter 只带 5 层精简上下文，Token 消耗更少、回复更快
• Worker 多出 6 层赋予工具调用、策略执行、审批等完整权限，两者天然实现权限隔离
产品设计 让对话更像真人
核心
人物活动轨迹模拟

传统 AI 没有"生活"。你问"今天干嘛了"，它会现编一个回答，没有一致性。

MyClaw 设计了五级生活线体系：Life Plan → Year Plan → Month Plan → Week Plan → Day Plan。从人生阶段一直分解到每日日程：

Life Plan（人生阶段：败光家产前 → 败光家产后 → 给龙二打工） └─ Year Plan（20 岁，当少爷） └─ Month Plan（春天的日常） └─ Week Plan（这周去镇上赌钱） └─ Day Plan（今天的具体行程）

Worker Agent 每天凌晨 3:00 自动运行，根据当前阶段规划当天生活，推进剧情事件，生成日志。这样：

  • 即使不聊天，角色也在真实地活着
  • 第二天用户问"昨天干嘛了"，角色有真实内容可说，不是现编的
  • 角色的生活是连贯的——昨天去了赌场，今天就不会突然在书房读书
自动触发与主动话题

大多数 AI 的对话模式是"你问他答"——你主动找它，它才回应。关掉对话窗口，它就"消失"了。

MyClaw 的设计目标之一是让角色也会主动找你。每天 Worker 更新角色第二天的生活计划时，会同步设计第二天在哪些时间点、以什么理由主动给用户发消息。比如：

  • 早上 8:00 — "起了没？今天天气预报说要下雨，带伞。"
  • 中午 12:30 — "中午吃了啥？我今儿在镇上下馆子，点了个红烧肉。"
  • 晚上 22:00 — "还不睡？明天不是要早起吗。"

这套主动推送机制由 Heartbeat 定时扫描 驱动，支持三种触发策略：

  • 定时触发 — 预设时间点（起床提醒、午饭时间、睡前）
  • 空闲触发 — 用户超过一段时间没说话，角色主动问一句
  • 被动失效 — 一个话题自然衰减后，平滑过渡到新话题

对话过程中，结合生活线状态与上下文，在话题自然衰减时平滑过渡，避免生硬的"对了，我还有个事要说"。目标是模拟真人聊天的联想式话题跳跃——不是机械地切换，而是顺着当前话题自然地想到另一件事。

这套机制完成了从被动响应到主动服务的范式升级。

时间感知与空间锚定

AI 对话最容易露馅的地方是——没有时空坐标。

你问"吃饭了吗"，它可能回答"还没呢，刚下班"——但上一句它才说在沙发上躺着。这就是时空割裂感。

MyClaw 构建了角色的时间认知模型和空间锚定系统：

  • 时间感知 — 对话中自然体现时间流逝感："刚在开车"、"等了你有二十分钟了"、"昨晚去看了一部电影"
  • 空间锚定 — 角色知道自己在哪，切换时合理："刚从外面回来"、"在书房看书"、"路过你公司楼下"

两套系统配合，让角色行为在时空维度上自洽——不会出现"上一秒在做饭，下一秒说在开会"的割裂感。用户能感觉到这是一个真正"活着"的人，而不是一个问答机器人。

小说人物迁移

这是获取角色的捷径——不用手动写人物设定，直接从小说的文本描述中提取。

系统自动完成三步：

  • 提取性格（人物档案）——脾气、说话方式、价值观
  • 提取经历与社交关系（动线）——从出生到结局的关键事件、认识的人、不知道的事
  • 时间线映射——把小说的虚构时间映射到真实世界时间轴

以《活着》中的福贵为例，过程是这样的：

小说原文 → SKILL 提取 ├─ soul.md——福贵 20 岁，徐家少爷，浪荡好赌，最近刚娶了家珍 ├─ timeline-life.md——第一阶段：少爷时期。【关键事件待触发】龙二设赌局 └─ "他不知道的事"——家珍已经怀了有庆

提取完成后，这个角色就自动开始他的"真实生活"——第二天 Worker 会推进他的日常，他在镇上赌钱、被爹骂、晚上回家。用户不需要手动捏人，直接跟小说人物聊天即可。

其他亮点
  • 安全体系：纵深防御，五层防护。
    第一层，PathGuard（路径围栏） — 文件操作不越界。把路径解析为真实路径后判断是否在项目根目录内，防止 LLM 通过绝对路径、.. 或软链接访问外部文件。
    第二层，CommandGuard（命令黑名单） — 正则会匹配拦截 sudo、rm -rf /、mkfs、Fork Bomb 等破坏性命令，减少 HITL 弹窗骚扰。
    第三层，NetworkPolicy（网络策略） — 防 SSRF。Scheme 白名单（仅 http/https）+ DNS 解析拦截内网地址 + Token Bucket 限流每分钟 30 次。
    第四层，浏览器安全 — 正则匹配敏感页面（bank、银行、Settings、设置），写入操作强制单步审批。
    第五层，HITL 人工审批 — 最核心一层。写入类操作需审批，提供批准、全部放行、拒绝、跳过、修改参数后执行五种操作。
  • 微信绑定：集成微信渠道接入，支持通过微信与 AI 对话，打通 IM 与 CLI 双端交互，用户可在微信端接收主动推送消息与定时提醒，扩展 AI 的可触达场景。
    需要注意微信对第三方机器人管控很严，个人号容易被风控封号，主动推送频率过高也会触发限制。目前是用 Agent 直接调用微信接口实现的，偏向实验性质，生产环境建议改用企业微信或公众号渠道。
基础能力
  • 多模型支持 — 支持 GLM / DeepSeek / SiliconFlow / StepFun / Kimi 多种 LLM，启动后可通过 /model 动态切换
  • ReAct 循环 — AI 的思考-调用-反思核心循环，支持多轮工具调用和并行执行
  • MCP 协议集成 — 通过 MCP 连接外部工具和服务（文件系统、浏览器等），启动时自动加载
  • Web 搜索与抓取 — 支持联网搜索（智谱/SerpAPI/SearXNG）和网页内容提取
  • Chrome DevTools 集成 — 连接本机 Chrome，支持页面查看和浏览器自动化
  • Skill 系统 — 可插拔能力模块，支持动态加载。内置 4 个 Skill：web-access（联网抓取）、daily-simulation（日常模拟）、scheduled-task（定时任务规则）、character-extraction（人物提取）
  • Inline 渲染器 — 终端内灰度显示思考过程，工具调用可按 Ctrl+O 折叠展开
  • 命令面板与 Tab 补全 — / 触发命令，支持 Tab 补全
  • 快照与回滚 — 操作前自动创建 git 快照，支持撤销
  • 操作审计 — 危险操作记录到 JSONL 日志，可追溯
  • PathGuard 路径围栏 — 自动校验文件路径，防止越界访问
