企业知识库 Agentic RAG

项目来源 为什么做这个项目
核心优化 全链路优化
1
文本分块优化
为什么需要分块

向量模型有长度限制、长文本语义太杂检索不准、大模型有窗口限制。

两级四层语义分块策略

第一层父块：流式解析，达到 parentChunkSize（1MB）自动切割，避免整个文章加载到内存导致 OOM。

第二层子块（四级语义切割）：将父块按 chunkSize 512 Token 分割为子切片，优先按段落（\n\n）→ 过长按句子 → 再长按词语（HanLP 分词）→ 兜底按字符分割，全程保证语义完整性。

重叠分块机制

每个分块前后保留上下文冗余，Embedding 模型处理时不会遗漏边界关键信息，召回率提升明显。

代码文件分块（AST 分析）

使用 JavaParser 做 AST 分析，把源码解析成语法树，提取类和方法的起止行号，再根据行号提取对应文本。类 chunk 只取前 5 行签名，保持语义完整性。

同时通过 AST 提取五种关系（extends / implements / imports / contains / calls）构建关系图谱，存在 MySQL 中，支持 Agent 执行代码任务时查询。

2
检索优化
文件上传流程

前端按 5MB 分片上传，同时计算文件 MD5。上传前通过 Redis Bitmap 校验防重复。所有分片上传承诺完成后，后端校验分片完整性，调用 MinIO 合并，更新 MySQL 状态为已完成，发送 RocketMQ 消息异步触发解析、分块、向量化。

异常处理：幂等设计防重复、断点续传、分片重试、MD5 校验完整性、定时任务核对 MinIO 与 MySQL 状态保证最终一致。

问题改写策略

五种策略分层组合：简单改写（消除代词、补充隐含信息）、问题拆解（多问题分开查）、HyDE（LLM 生成假设答案后检索）、多角度扩写（3-5 种表述分别检索后合并去重）、后退提问（抽象到通用原理层面再检索）。

混合检索流程
1. 问题改写（不同问题不同策略）
2. 调用 Embedding 模型转 2048 维查询向量
3. IK 分词器对问题进行中文分词
4. 向量检索（knn 查询，余弦相似度），召回候选池
5. BM25 关键词检索，计算精确相关性分数
6. RRF 融合排序，合并两路结果
7. ReRank 精细排序，取前 k 篇
3
问题理解优化
智能路由

通过关键词/正则匹配对问题分类，简单问题（问候、数学等）直接回答，跳过向量检索，提升速度并降低 Embedding API 调用成本。

多跳推理

针对复杂问题分解为子查询链，每步基于前序结论动态构建新查询。例如"程愚瀚和鲁迅的关系"→ 先查程愚瀚（电子科大学生）→ 再查电子科大的人才 → 关联到鲁迅。弥补向量 RAG 在全局性问题和多跳推理上的短板。

4
生成优化
幻觉控制

Prompt 优化 + 结构化输出约束 + 低分拒答 + 幻觉检测机制，确保生成内容忠实于检索结果。

问答流程

用户提问后，前置决策判断是否需要检索。需要则执行问题改写 → 向量检索 → 低分时触发联网搜索兜底。送入大模型前自动抽取偏好信息、拉取历史上下文、记忆压缩。

全链路加入双层内容审核：生成前风控拦截，生成后通过 RAGAS 校验忠实性，综合判定是否需要多轮迭代补全重生成。

技术原理 核心技术原理与选型
向量检索原理
长度代表内容的多少、信息量的大小；角度代表语义。
余弦相似度：计算两个向量的夹角，夹角越小越相似。只看方向。
欧式距离：计算两个向量的距离，距离越小越近。
点积：计算方向一致性与长度的乘积（l₁·l₂·cosθ），结果越大越相似。既看方向又看长度，但需要归一化，否则长度差异会掩盖角度。
BM25 原理

ES 默认的关键词相关性算法，根据词频、文档长度计算匹配度。通过两个参数控制：

k1：越大，关键词频率对匹配度影响越大
b：越大，对长文档的惩罚越重（相同词语出现次数，文章越长匹配度越低）
RRF 重排序原理

在多路排序列表中忽略不同排序方式的分数，只看排名。好处是兼容不同打分体系，缺点是完全忽略分数差距（如相关性 100 和 1 的文档在排名上差距不大）。

• 倒数打分：每个列表里按位置算分 1/(k+rank)
• 分数累加：同一文档在所有列表中得分求和
• 重排输出：按总分排序得融合结果
k：平滑常数，默认 60。k 越大越平均，重"出现次数"；k 越小第一名优势越强，适合强头部场景。
ReRank 原理

Embedding 模型（Bi-Encoder）：分别计算 Query 和文档的向量后对比相似度。速度快、可全库搜索，但只做了表面匹配，没有深度交互。

Rerank 模型（Cross-Encoder）：把 Query 和文档拼接在一起直接分析词间关系，输出 0~1 相关性分数。更深层理解，但速度慢、开销大。

先 Embedding 搜广度，再 Rerank 做精细化筛选。

技术选型

遵循不要钱、能快速跑起来为第一原则。

Embedding 框架：Spring AI 的 EmbeddingModel，适配主流向量模型，配置即用
向量模型：阿里 text-embedding-v4（MTEB 评测第三，国内使用首选）
向量模型（免费）：豆包
存储检索：Elasticsearch（最早使用）、Milvus / Pinecone 等专业向量数据库
Rerank：BGE-Rerank / Cohere Rerank 等
其他亮点
权限控制

基于 Spring Security 搭建三层过滤器链：

第一层：提取、校验 Token，封装用户信息到 SecurityContextHolder
第二层：按接口规则完成角色级权限校验
第三层：加载组织标签用于数据级权限控制
搜索模块自动构建带权限条件的 ES 查询，强制只返回用户有权查看的数据
MCP Server

实现 MCP Server，支持外部 AI Agent 通过标准化协议接入查询知识库，打破工具间的数据孤岛。
