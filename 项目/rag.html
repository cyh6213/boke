混合检索流程
1
问题改写
首先，问题改写（不同问题，不同策略）
2
向量化
调用 BGE 的 embedding 模型（BAAI/bge-m3），将用户的问题转换成 dense 向量。
3
稀疏向量化
同时通过 BGE 生成 sparse 向量，替代传统 BM25 关键词检索，实现更精确的关键词级别匹配。
4
向量搜索（dense）
用 dense 向量去 Milvus 中执行 HNSW 索引搜索，进行余弦相似度计算，找出语义最相近的一大批内容（比如 k*30=300 篇）。
5
稀疏搜索（sparse）
用 sparse 向量做关键词级别的精确匹配，综合评估每个文档与查询的命中程度，作为语义搜索的互补。
6
RRF 融合
通过RRF算法得出排名靠前的
7
Rerank 重排
在进行Rerank，最后选取前k篇。
问答流程
1
前置决策
用户提问接入后，首先通过前置内容风控拦截，包含两层防护：
提示词防护
拦截 Prompt 注入、越狱攻击、敏感指令、爬虫探测，保护 LLM 安全。
敏感数据脱敏
识别手机号 / 身份证 / 银行卡 / 邮箱 / 地址后，支持自动掩码替换（如 138****1234），返回脱敏后内容。
2
分类路由
对问题进行分类后，不同类型走不同路径：
简单问题
省略向量查询，提升响应速度。
多跳 / 复杂问题
在检索策略、改写深度、上下文组装方式和生成策略四个维度进行针对性的优化，走不同的路径。
3
检索拒答
完成改写后执行混合检索，如果检索结果评分没有达到阈值，或者没有检索出结果，就直接回复没有资料，而不是让大模型对着不相干的内容硬编，降低幻觉发生概率。
4
生成审核
模型输出结果后，会通过RAGAS校验回答的忠实性，综合判定是否需要多轮迭代补全重生成。

分片优化
为什么分片？
主要有三个原因：
  • 向量模型有长度限制
  • 长文本语义太杂、检索不准。
  • 丢给 大模型回答时，依然有窗口限制。
我们的分片策略？
—— 两层四级语义分块策略
第一层 · 父块
流式解析，达到 parentChunkSize（1MB） 自动切割，而不是把整个文章加载到内存再分块，避免 OOM。
第二层 · 子块（四级语义切割）
将父块文本按 chunkSize 512 TOKEN 分割为更小的子切片
  • 优先按段落分割（\n\n）
  • 段落过长按句子分割
  • 句子过长按词语分割（HanLP 分词，不拆词）
  • 最后兜底按字符分割全程保证语义完整性，不会把完整句子、词语强行切断。
重叠分块
最后，我还让每个分块的前后都有一部分"上下文冗余"，这样， embedding 模型在处理时就不会遗漏关键信息。
其他分片策略？
  • 固定大小
  • 按照层级（章节、内容），适合结构清晰的文档
  • 按照语义
  • 特殊内容：代码（以函数为单位）、表格（要保留表头，可以转化为json）
  • 父子：检索用小分片，返回大分片
  • 语义补全：用LLM给每一个分片都生成一个上下文，代价就是成本高
代码分片策略 代码专用
切片部分我用 JavaParser 类做 AST 分析，把源代码解析成语法树，然后遍历树节点提取类和方法的起止行号，再根据行号从原始代码中提取对应文本。
这种方式的优势在于能够保持代码的语义完整性，不会像按字符数分段那样切断方法或类，同时还能获取类名、方法名等元数据，为后续的代码检索和分析提供更丰富的信息。
类 chunk 只提取前 5 行作为类签名，不会把整个类的代码都塞进去。长方法目前没有做二次切分，实际项目中超过这个长度的方法比较少见，真要改善可以按内部逻辑块继续拆分。
问题改写的策略
问题改写有五种策略分层组合：
简单改写 — 处理口语化查询，消除代词补充隐含信息。
问题拆解 — 把多个问题拆解分开查询，还有对比类问题，A手机和B手机的性能对比，在我制定的策略就是分开查询然后
HyDE — 让LLM生成假设答案，用答案向量去检索，因为答案和专业文档的用词更接近，语义匹配度更高。
多角度扩写 — 把查询改写成3到5种表述分别检索后合并去重，覆盖不同作者的用词习惯。Query扩展补充相关关键词，扩大召回范围。
后退提问 — 把具体问题抽象到通用原理层面，先检索背景知识建立上下文，再检索具体细节。
幻觉优化 降低幻觉率

为了降低生成阶段的幻觉率，在生成环节进行了多项优化：Prompt 优化、结构化输出约束、低分拒答、幻觉检测机制，确保生成内容忠实于检索结果。

① Prompt 约束 — 让他没有的就说不知道
② 低分拒答 — 如果混合检索或者Rerank得出来的分数过低，就直接回复没有资料，而不是让大模型对着不相干的内容硬编。
③ 结构化输出 — 让 LLM 输出结构化的结论，每个结论都必须填写来自哪条参考资料。
④ RAGAS 核查 — 生成后再调用一次LLM核查或者使用RAGAS来做核查，来检查忠实性、答案相关性。
⑤ 参数调低 — 把 temperature、top-p 等参数调低，让他回答的更严谨，避免跑偏。
智能路由 简单问题跳过检索

为了提升召回速度并降低 Embedding API 调用成本，通过智能路由对问题分类：简单问题（问候/数学等）通过关键词/正则匹配直接回答，跳过向量检索，提升召回速度并减少 API 调用。

路由策略（你是怎么对问题进行分类的）
系统采用三层决策机制判定问题类型和复杂度。
第一层 · 规则引擎
基于关键词和正则匹配快速过滤闲聊、纯计算类问题，这些直接走模型回答，不查向量库。
第二层 · 语义分类器
用 Spring AI Alibaba 的 QuestionClassifierNode 做查询分类，我在里面定义了五种维度包括是否需要检索、单跳还是多跳、问题类型是技术操作还是对比分析。
第三层 · 实体关系检测
提取查询中的实体数量和关系复杂度，计算复杂度分数，超过阈值自动判定为多跳查询。
处理策略举例

不同类型查询的处理差异体现在检索策略、改写深度、上下文组装方式和生成策略四个维度。

● 单跳事实查询
用高阈值精确召回1到3个片段。除了口语化处理之外，聚焦关键词提取和指代消除。生成时严格限定在召回内容内，必须标注来源，禁止引入外部知识。
● 多跳推理查询（ReAct）（程愚瀚和鲁迅的关系是什么）
需要执行多次向量检索，要分解为子查询链，每步基于前序结论动态构建新查询。程愚瀚是电子科大的学生，然后电子科大又有什么人才，然后就可以和鲁迅联系上了。检索时用较低阈值（0.6）做宽召回。生成是要求标注推理步骤。
● 技术操作查询
重点在生成策略，要求他使用分步骤模版。
● 对比查询
要执行两次独立检索分别放入A池和B池。单边缺失标注信息不足，禁止用模型知识填补空白。

上传优化

基于 Redis Bitmap 实现并发分片上传与状态追踪，结合 RocketMQ 异步解析解耦，大文件上传响应速度提升 65%。

流程：
① 用户上传文件，前端按 5MB 分片上传
② 后端校验文件类型，存入 MinIO，Redis Bitmap 记录进度，持久化到 MySQL
③ 所有分片完成后，校验完整性，MinIO 合并文件，MySQL 更新状态
④ 发送 RocketMQ 消息，异步触发文档解析、向量化、写入 ES
策略：
1. 幂等设计 — 针对重复上传，每次上传前查 Redis 标记
2. 断点续传 — 针对网络中断，重传自动跳过已上传部分
3. 重试机制 — 针对单个分片上传失败，利用前端内置重试
4. 完整性校验 — 针对数据完整性，后端算 MD5 判断
5. 最终一致 — 针对合并成功但更新 MySQL 前宕机，定时任务核对自动修复
权限控制

基于 RBAC + OrgTag + Spring Security + JWT 实现多级权限控制与双令牌认证，支持令牌无感刷新，严格确保用户仅能检索到自身权限范围内的文档。

登陆控制（双令牌）
Access Token 核心通行证，有效期 1 小时，缩短时间窗口，减小泄漏风险
Refresh Token 长期凭证，有效期 7 天，泄漏也无法直接调用接口，可通过 Redis 快速吊销
自动刷新：当 Access Token 距离过期 < 20 分钟时访问，自动刷新令牌
两层权限控制
角色控制能否进入网页，标签决定能否访问文件。基于 Spring Security 搭建过滤器链：
1 提取校验 Token，封装用户信息到 SecurityContextHolder
2 角色级接口权限校验，判断是否有权访问当前接口
3 加载组织标签，存入请求上下文，用于数据级权限控制
4 搜索时自动构建带权限条件的 ES 查询，强制只返回有权查看的数据
MCP

实现 MCP Server，支持外部 AI Agent 通过标准化协议接入查询知识库，打破工具间的数据孤岛。员工无需直接打开本系统，可在任意支持 MCP 的 Agent 平台中调用知识库。

场景：为什么转成 MCP？
公司内部员工日常使用各类 AI Agent（如 Cursor、Dify、Claude Desktop 等）来完成开发、文档编写等任务。这些 Agent 需要访问企业内部的私域知识，但不可能让每个员工都打开本系统手动查询。同时，本系统的问答接口消耗 Token，更适合作为"资料提取层"而非回答层。

因此把 RAG 系统封装为 MCP Server —— 只负责按权限提取文档片段并返回原文，不承担回答生成。员工的外部 Agent 拿到原文后，自行组装回答。这样既节省了 Token 开销，又让知识库能力被任意 Agent 生态复用。
实现：如何让别人看见？
基于 Spring AI Alibaba 的 MCP Server Starter，在现有项目中增加一个 spring-ai-mcp-server-webmvc 依赖即可。项目启动后自动暴露一个 MCP 端点，对外提供两个标准方法：listTools（告知客户端有哪些可用工具）和 callTool（执行实际检索）。

员工在 Agent 端的 MCP 配置文件中填入本服务的地址（如 http://公司内网IP:8080/mcp），Agent 就能自动发现 searchKnowledge 工具并调用。整个过程就像在应用商店安装插件一样，零开发、即配即用。
权限：外部调用如何隔离？
MCP 协议本身没有认证机制，权限完全在应用层实现。核心思路是复刻网页端的权限模型：

员工外部 Agent 调用时，通过 MCP 请求参数携带员工的 Access Token（由 Agent 平台在请求时自动注入）。MCP Server 内部的 searchKnowledge 方法第一件事就是校验 Token 有效性、解析出用户身份，然后走和网页端完全一致的过滤器链 —— 角色级接口校验、组织标签加载、ES 权限查询。最终返回的数据已经过权限过滤，员工只能看到自己有权限的文档。

这套逻辑不依赖 HTTP 会话或 SecurityContextHolder，完全从参数传入，天然适配无状态的 MCP 协议。同时支持通过 Redis 快速吊销 Token，泄漏后即时失效。

核心技术原理 检索核心技术详解
向量检索的原理
长度代表内容的多少，信息量的大小
角度代表语义
余弦相似度：计算两个向量的夹角，夹角越小，就越近。只看方向
欧式距离：计算两个向量的距离，距离越小，就越近
点积：计算两个向量方向一致性与长度的数值（l1*l2*cos），结果越大通常表示向量越相似。既看方向，又看长度。需要归一化，也就是 长度差异特别大时，有的向量明明角度很大，但是可能被长度掩盖
BM25 原理

ES 默认的关键词相关性算法，根据词频、文档长度计算匹配度。会通过两个关键词控制匹配度：

k1：越大，关键词频率对匹配度影响就越大
b：越大，对长文档的惩罚就会越重（相同词语出现次数，文章越长，匹配度月底）
稀疏索引、稠密索引与关键词检索的区别
● 稠密向量：是把文本压缩成一个固定维度（如 2048 维）的浮点数向量，每个维度"均匀分摊"了语义信息，你很难说哪一个维度对应哪个词。它的优势是能理解"猫 vs 狗是相似的"这种语义相似度，缺点是精确匹配差——搜"鲁迅"可能把"周树人"拉进来，但搜"红烧肉"不会把和它同在一个文档里的"糖色"关联上。
● 稀疏向量：是把文本映射成一个超高维（几万到几十万维）的向量，但大部分维度是 0，只有少数维度有值。每一个维度对应一个具体的词或 token，值表示这个词的重要性（类似 TF-IDF 的权重）。所以它既保留了关键词级别的精确匹配能力（因为 apple 这个词的维度就是 apple，不会跟 fruit 混在一起），又具备向量空间的距离计算能力（可以计算"猫狗"在高维空间的余弦相似度）。比 BM25 强的地方在于它可以用学习到的词权重，而不是纯统计的 IDF。
● 关键词检索（BM25）：基于词频统计的精确匹配算法，只计算查询词与文档中相同词的匹配程度。速度快、可解释性强，但无法理解同义词、近义词等语义关系。
稀疏向量是稠密向量和关键词的一个"中间态"：
下接关键词的精确性（每个维度对应具体词）
上接稠密向量的语义理解（可做向量空间的距离计算）
这也是为什么双路召回用稠密 + 稀疏搭配效果最好：稠密负责语义泛化，稀疏负责精确命中。
RRF 重排序
在多路排序列表中，忽略不同排序方式的分数，只看排名的一种排序方式。
好处：可以忽略不同排序方式的分数
缺点：因为他忽略了分数，所以他就不包含分数，比如一个文字的相关性是100，第二名是1，但是根据排名来说，一个是1/1，一个是1/2，他们的相关性差距就不大。
● 倒数打分：每个列表里，按位置算分 1/(k+rank)；
● 分数累加：同一文档在所有列表中的得分求和；
● 重排输出：按总分从高到低排序，得到融合结果
● k：平滑常数，默认 60；k越大，得分越平均， 更重视 "出现次数" 而非 "靠前程度"；k越小， 第一名优势越强，适合强头部场景。
ReRank 原理

Embedding 模型用的Bi‑Encoder（双向编码器）：分别计算 Query 和文档的向量，然后对比相似度。好处是快、可以全库搜索。但是只看到了表面的匹配，没有做深度交互。

Rerank 模型用的是Cross‑Encoder（交叉编码器）：它会把Query 和文档拼接在一起，直接分析每个词之间的关系，而不是先生成向量在比较，就让他能够更深度的理解Query 和文档的关系。最后输出输出一个0 到 1 之间的相关性分数。代价：速度慢，开销大。

所以一般是先用enmbedding搜广度，再用Rerank精细化。
