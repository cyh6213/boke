Project 01

企业知识库 Agentic RAG

项目来源 为什么做这个项目
核心优化 全链路优化
召回率优化 提升召回率

为了提升召回率，对整条流水线进行了全链路优化：上传阶段采用两级四层分片策略；检索阶段采用问题改写 + 双路召回（向量 + 关键词）；排序阶段采用 RNN 融合策略 + ReRank 重排。经过这些优化，Recall@5 从 68% 提升至 93%。

分片优化
为什么分片？
主要有三个原因：
● 向量模型有长度限制
● 长文本语义太杂、检索不准。
● 丢给 大模型回答时，依然有窗口限制。
我们的分片策略？
两层四级语义分块策略：
1. 第一层父块：流式解析，达到 parentChunkSize（1MB） 自动切割，而不是把整个文章加载到内存再分块，避免 OOM。
2. 第二层子块（四级语义切割）：将父块文本按 chunkSize 512 TOKEN 分割为更小的子切片
○ 优先按段落分割（\n\n）
○ 段落过长按句子分割
○ 句子过长按词语分割（HanLP 分词，不拆词）
○ 最后兜底按字符分割全程保证语义完整性，不会把完整句子、词语强行切断。

最后，我还计划引入重叠分块机制。我们让每个分块的前后都有一部分"上下文冗余"，这样， embedding 模型在处理时就不会遗漏关键信息，召回率提高了很多。

其他分片策略？
● 固定大小
● 按照层级（章节、内容），适合结构清晰的文档
● 按照语义
● 特殊内容：代码（以函数为单位）、表格（要保留表头，可以转化为json）
● 父子：检索用小分片，返回大分片
● 语义补全：用LLM给每一个分片都生成一个上下文，代价就是成本高
代码分片策略

切片部分我用 JavaParser 类做 AST 分析，把源代码解析成语法树，然后遍历树节点提取类和方法的起止行号，再根据行号从原始代码中提取对应文本。

这种方式的优势在于能够保持代码的语义完整性，不会像按字符数分段那样切断方法或类，同时还能获取类名、方法名等元数据，为后续的代码检索和分析提供更丰富的信息。

类 chunk 只提取前 5 行作为类签名，不会把整个类的代码都塞进去。长方法目前没有做二次切分，实际项目中超过这个长度的方法比较少见，真要改善可以按内部逻辑块继续拆分。

问题改写的策略？
问题改写有五种策略分层组合。
● 简单改写处理口语化查询，消除代词补充隐含信息。
● 问题拆解把多个问题拆解分开查询，还有对比类问题，A手机和B手机的性能对比，在我制定的策略就是分开查询然后
● HyDE让LLM生成假设答案，用答案向量去检索，因为答案和专业文档的用词更接近，语义匹配度更高。
● 多角度扩写把查询改写成3到5种表述分别检索后合并去重，覆盖不同作者的用词习惯。Query扩展补充相关关键词，扩大召回范围。
● 后退提问把具体问题抽象到通用原理层面，先检索背景知识建立上下文，再检索具体细节。
幻觉优化 降低幻觉率

为了降低生成阶段的幻觉率，在生成环节进行了多项优化：Prompt 优化、结构化输出约束、低分拒答、幻觉检测机制，确保生成内容忠实于检索结果。

● 首先是prompt约束，让他没有的就说不知道
● 其次是如果混合检索或者Rerank得出来的分数过低，就直接回复没有资料，而不是让大模型对着不相干的内容硬编。
● 其次是可以让 LLM 输出结构化的结论，每个结论都必须填写来自哪条参考资料。
● 其次可以是生成后再调用一次LLM核查或者使用RAGAS来做核查，来检查忠实性、答案相关性。
● 最后是可以把tempreture、top-p等参数调低，让他回答的更严谨，避免跑偏。
智能路由 简单问题跳过检索

为了提升召回速度并降低 Embedding API 调用成本，通过智能路由对问题分类：简单问题（问候/数学等）通过关键词/正则匹配直接回答，跳过向量检索，提升召回速度并减少 API 调用（加上数据）。

路由策略（你是怎么对问题进行分类的）
系统采用三层决策机制判定问题类型和复杂度。
第一层是规则引擎，基于关键词和正则匹配快速过滤闲聊、纯计算类问题，这些直接走模型回答，不查向量库。
第二层是语义分类器，用 Spring AI Alibaba 的 QuestionClassifierNode 做查询分类，我在里面定义了五种维度包括是否需要检索、单跳还是多跳、问题类型是技术操作还是对比分析。
第三层是实体关系检测，提取查询中的实体数量和关系复杂度，计算复杂度分数，超过阈值自动判定为多跳查询。
处理策略举例

不同类型查询的处理差异体现在检索策略、改写深度、上下文组装方式和生成策略四个维度。

● 单跳事实查询
用高阈值精确召回1到3个片段。除了口语化处理之外，聚焦关键词提取和指代消除。生成时严格限定在召回内容内，必须标注来源，禁止引入外部知识。
● 多跳推理查询（ReAct）（程愚瀚和鲁迅的关系是什么）
需要执行多次向量检索，要分解为子查询链，每步基于前序结论动态构建新查询。程愚瀚是电子科大的学生，然后电子科大又有什么人才，然后就可以和鲁迅联系上了。检索时用较低阈值（0.6）做宽召回。生成是要求标注推理步骤。
● 技术操作查询
重点在生成策略，要求他使用分步骤模版。
● 对比查询
要执行两次独立检索分别放入A池和B池。单边缺失标注信息不足，禁止用模型知识填补空白。
多跳推理 弥补向量 RAG 的全局性问题短板

针对复杂问题（如多跳推理查询），将其分解为子查询链，每步基于前序结论动态构建新查询。例如"程愚瀚和鲁迅的关系"→ 先查程愚瀚（电子科大学生）→ 再查电子科大的人才 → 关联到鲁迅。这一策略弥补了向量 RAG 相较于 GraphRAG 在全局性问题、多跳推理等需要逻辑链条的复杂问题上的短板。

流程 核心流程
混合检索流程
● 首先，问题改写（不同问题，不同策略）
● 再调用豆包的 embedding 模型，将用户的问题转换成一个 2048 维的"查询向量"。
● 以及ik分词器，将问题分词（默认情况下，ES 对中文就是「一个字一个词」拆分）。
● 接着，向量搜索。用这个向量去和 ES 中所有文档的向量做对比，找出意思最相近的一大批内容（比如 k*30=300 篇）。执行 knn 查询，进行余弦相似度计算。
● 接着，关键词查询。使用 BM25 算法，综合评估每个文档中关键词的质量（词频、重要性、文档长度等），并计算出一个精确的关键词相关性分数
● 通过RRF算法得出排名靠前的（但是权值一样）
● 在进行Rerank，最后选取前k篇。
问答流程

我设计的 RAG 系统，在用户提问接入后，首先会进行前置决策判断，判定当前问题是否需要调取向量库检索和工具（以及对问题进行分类，有不同的执行策略）需要检索则执行问题改写优化，提升查询语义准确性。

完成改写后执行向量数据库检索，如果检索结果评分没有达到阈值，或者没有检索出结果，会自动触发联网搜索作为信息兜底补充，同时在prompt做提示。

在送入大模型生成前，自动抽取并学习用户提问中的偏好信息，同步拉取历史对话上下文，完成记忆压缩精简，控制上下文长度与 token 消耗。

全链路加入双层内容审核机制，大模型生成回答前完成前置内容风控拦截，模型输出结果后，会统一开展后置评估，通过RAGAS校验回答的忠实性，综合判定是否需要多轮迭代补全重生成。

其他 其他亮点
上传优化

基于 Redis Bitmap 实现并发分片上传与状态追踪，结合 RocketMQ 异步解析解耦，大文件上传响应速度提升 65%。

流程：
用户上传文件时，前端按 5MB 大小分片上传；后端先校验文件类型，将分片存入 MinIO，同时用Redis Bitmap 记录分片上传进度，并把分片信息持久化到 MySQL。所有分片上传完成后，后端先校验分片完整性，再调用 MinIO 合并文件，合并成功后更新 MySQL 文件状态为已完成，最后发送RocketMQ 消息异步触发文档解析、向量化、写入 ES，让前端快速响应
策略：
首先是针对重复上传，做了幂等设计，每次上传前都会查 Redis 标记；
针对网络中断，支持断点续传，中断后重传会根据Redis上传记录自动跳过已上传部分，不会重复上传；
针对单个分片的上传失败，利用前端内置的重试机制；
针对数据的完整性，后端会算MD5值来判断
其次是，针对合并成功但更新 MySQL 前宕机的问题，通过定时任务（当用户发现文件很久都没有上传成功，点击刷新时，就回去核对状态）核对 MinIO 与 MySQL 状态，自动修复未更新的文件记录，保证最终一致。
权限控制

基于 RBAC + OrgTag + Spring Security + JWT 实现多级权限控制与双令牌认证，支持令牌无感刷新，严格确保用户仅能检索到自身权限范围内的文档。

登陆控制（双令牌）
Access Token + Refresh Token
Access Token：核心通行证，有效期为1小时，缩短时间窗口，减小token泄漏带来的风险
Refresh Token：获取新 Access Token 的 "长期凭证"，有效期为7天（7天内避免重复登陆），即便泄露，攻击者也无法直接调用业务接口（需先换 Access Token），且你可通过 Redis 快速吊销 Refresh Token
当 用离过期 < 20 分钟时的Access Token访问就会主动刷新。
两层权限控制（角色控制能否进入网页、标签决定能否访问文件）
基于 Spring Security 搭建了一个过滤器链
● 第一层Filter 从请求头提取、校验、解析 Token，将合法用户信息封装到 SecurityContextHolder
● 第二层Filter：根据 SecurityConfig 中配置的接口访问规则完成角色级接口权限校验，判断用户是否有权访问当前接口。
● 第三场Filter：从数据库加载当前用户的组织标签，存入请求上下文，用于后续数据级权限控制。
● 进入搜索模块后，系统自动构建带权限条件的 Elasticsearch 查询，强制只返回用户有权查看的数据：自己的私有文档、公开文档、所属组织标签下的文档

原理 核心技术原理与选型
向量检索的原理
长度代表内容的多少，信息量的大小
角度代表语义
● 余弦相似度：计算两个向量的夹角，夹角越小，就越近。只看方向
● 欧式距离：计算两个向量的距离，距离越小，就越近
● 点积：计算两个向量方向一致性与长度的数值（l1*l2*cos），结果越大通常表示向量越相似。既看方向，又看长度。需要归一化，也就是 长度差异特别大时，有的向量明明角度很大，但是可能被长度掩盖
BM25原理

ES 默认的关键词相关性算法，根据词频、文档长度计算匹配度。会通过两个关键词控制匹配度：

k1：越大，关键词频率对匹配度影响就越大
b：越大，对长文档的惩罚就会越重（相同词语出现次数，文章越长，匹配度月底）
RRF重排序

作用：在多路排序列表中，忽略不同排序方式的分数，只看排名的一种排序方式。好处是可以忽略不同排序方式的分数，缺点也是因为他忽略了分数，所以他就不包含分数，比如一个文字的相关性是100，第二名是1，但是根据排名来说，他们的相关性差距就不大。

● 倒数打分：每个列表里，按位置算分 1/(k+rank)；
● 分数累加：同一文档在所有列表中的得分求和；
● 重排输出：按总分从高到低排序，得到融合结果
● k：平滑常数，默认 60；k越大，得分越平均， 更重视 "出现次数" 而非 "靠前程度"；k越小， 第一名优势越强，适合强头部场景。
Rerank原理

Embedding 模型用的Bi‑Encoder（双向编码器）：分别计算 Query 和文档的向量，然后对比相似度。好处是快、可以全库搜索。但是只看到了表面的匹配，没有做深度交互。

Rerank 模型用的是Cross‑Encoder（交叉编码器）：它会把Query 和文档拼接在一起，直接分析每个词之间的关系，而不是先生成向量在比较，就让他能够更深度的理解Query 和文档的关系。最后输出输出一个0 到 1 之间的相关性分数。代价：速度慢，开销大。

所以一般是先用enmbedding搜广度，再用Rerank精细化。、
