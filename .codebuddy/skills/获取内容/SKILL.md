# 获取内容助手

## 概述

这个 skill 定义了在回答用户问题之前如何系统性地获取信息。它不是简单的"搜一下然后回答"，而是一套从宽到窄、从概述到原文、多来源交叉验证的信息获取策略。

**核心原则：优先获取一手信息源，二手总结仅作辅助参考。**

## 信息来源分层（按可信度和信息密度排序）

### 第一层：官方一手源（最高优先级）

这类来源是信息的原始出处，可信度最高。包括：

- **官方技术博客**：如 LangChain Blog、Anthropic Blog、OpenAI Blog、Meta AI Blog、Google AI Blog 等
- **原始论文（arXiv）**：学术研究的源头，如 `arxiv.org`
- **官方文档和规范**：如 `docs.langchain.com`、`code.claude.com/docs`、`modelcontextprotocol.io`
- **技术领袖的原始博客**：如 Martin Fowler 的博客（martinfowler.com）、Mitchell Hashimoto 的博客等
- **GitHub 官方仓库**：项目的 README、文档、源码注释

搜索策略：
- 优先搜索 `site:langchain.com "topic"`、`site:anthropic.com "topic"`、`site:arxiv.org "topic"`
- 用 `"topic" site:github.com` 搜索官方仓库
- 用 `"topic" blog` 搜索官方技术博客

### 第二层：系统性分析文章

这类文章通常由独立研究者或技术博主撰写，对某个主题做了完整的归纳整理，信息密度高。

搜索策略：
- 英文：搜索 `"topic" architecture internals explained` 或 `"topic" complete guide`
- 优先选择标注了信息来源可信度分级的文章
- 优先选择引用了具体源码常量、函数名的文章（细节越多可信度越高）
- **优先选择英文一手分析**（如 dev.to、medium 上的深度技术文章），而非中文二手总结

### 第三层：源码逆向分析文章

这类文章基于实际源码进行逐行分析，提供函数名、常量值、文件路径等精确细节。

搜索策略：
- 用源码文件路径作为搜索锚点
- 用函数名作为精确锚点
- 用常量名作为精确锚点
- 中文技术社区（CSDN、博客园、掘金、SegmentFault）在这类内容上通常比英文社区更深入

### 第四层：中文二手总结（最低优先级，仅作辅助参考）

这类来源包括 CSDN、掘金、知乎等中文平台上由个人撰写的总结性文章。它们通常是第一、二层信息源的翻译或二次加工，可能存在信息失真、过时、或加入个人理解偏差的问题。

使用策略：
- **仅用于交叉验证**，不作为主要信息来源
- 优先找到原文出处，而非依赖中文总结
- 如果中文总结与一手源描述不一致，**以一手源为准**
- 仅在无法访问一手源（如页面 403、需登录）时，作为备选参考

### 第五层：Prompt 和源码原文仓库

直接从编译产物或泄漏源码中提取的原始文本，100% 准确但组织方式不友好。

搜索策略：
- GitHub 上搜索 `system-prompts` 仓库，按文件名定位具体内容
- 注意文件名可能与预期不同，先查看目录列表再读取
- 对于 prompt 原文，搜索 `"prompt keyword" site:github.com` 或 `"prompt keyword" gist`

## 多轮收敛搜索策略

不要期望一次搜索就能拿到全部信息。采用从宽到窄的收敛策略：

第一轮（宽泛关键词）：用主题名 + 架构/原理等通用词搜索，目标是找到概述性文章，建立全局认知。

第二轮（精确关键词）：从第一轮结果中提取具体术语（函数名、文件路径、常量名），用这些精确锚点进行第二次搜索，找到源码级细节。

第三轮（原文搜索）：如果还需要 prompt 原文或配置原文，用精确匹配的关键词片段加 `site:github.com` 搜索。

第四轮（交叉验证）：对关键事实用不同来源验证。如果两个来源对同一细节描述不一致，以引用了具体源码细节的来源为准。

## 中英文互补策略

- **英文一手源优先**：官方博客、原始论文、技术领袖博客是信息源头，优先获取
- **中文资料作补充**：中文技术社区（CSDN、博客园、掘金）的源码分析文章在细节层面有参考价值，但应追溯其引用的原文出处
- 对同一主题，至少读一篇官方/英文一手源，中文资料仅作交叉验证

## 交叉验证规则

单一来源可能出错，关键事实必须交叉验证：

- 执行顺序、常量值、阈值数字 → 至少两个独立来源确认
- 如果两个来源对同一细节描述不一致 → **以一手源（官方文档、原始论文、源码）为准**
- 如果所有来源都是一致的概述性描述（无源码引用） → 标注为"社区推断，未经源码验证"
- **中文二手总结与一手源矛盾时，一律以一手源为准**

## 输出规范

获取信息后，按以下原则整理输出：

- 直接回答用户的问题，不铺垫搜索过程
- 如果某个细节只从单一来源获取且无源码验证，在回答中体现不确定性
- 对无法获取到的关键信息，主动告知用户信息不完整
- **尽量标注信息来源**，让用户知道是来自官方一手源还是二手总结
