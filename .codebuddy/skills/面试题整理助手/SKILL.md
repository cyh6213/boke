# 面试题整理助手

## 概述

这个 Skill 负责管理用户的待解决面试题生命周期：接收问题 → 二级分类 → 识别已知 → 面试复盘 + 分析洞察 → 按时间优先推荐 → 提醒推进。

## 待完成目录结构

```
待完成/
├── 问题收件箱.md           # 快速倾倒问题（随意格式，复制粘贴即可）
├── 待解决问题.md           # 整理后的结构化问题（按日期+分类排列）
├── 待阅读/                 # 待阅读的文章/面经
└── 待办事项.md             # 非技术问答的待办
```

## 触发场景

### 场景一：添加新问题（收件箱模式）

用户可能说「加一个问题」或直接把问题粘贴到 `问题收件箱.md`。

**流程**：
1. 用户随意粘贴问题到 `问题收件箱.md`（无需格式化，支持列表、段落、面经原文等任意格式）
2. 用户说「整理」或「处理收件箱」
3. AI 读取收件箱，提取所有独立问题
4. 对每个问题确定二级分类、内部判定题型、判断是否与已有问题重复
5. 追加到 `待解决问题.md`（按日期分组，当天新增打上当天日期）
6. 在收件箱末尾添加 `[已处理 YYYY-MM-DD]` 标记
7. 输出本次添加了多少问题，当前总计多少，最老的问题积攒了多少天

### 场景二：专项整理

用户选「A. 技术问答归档 → 整理模式」时触发。

**流程**：
1. 读取 `待解决问题.md` 和 `待阅读/` 中的所有面经文件
2. 如果文件包含面试复盘内容（如「面试感受」「整体感觉」段落），先提取面试复盘
3. 提取所有问题，确定二级分类
4. 按已知/未知判定逻辑分级
5. 输出完整整理报告（含面试复盘 + 问题分级）
6. 用户确认后操作（移除已知、保留待学）

### 场景三：会话启动提醒

每次会话开始时自动执行。

**流程**：
1. 读取 `待解决问题.md`，统计待解决问题数量
2. 如果 ≥ 3 个且最老的问题超过 7 天，在「今天想做什么」询问后追加提醒

提醒格式：
```
📌 你有 X 个问题未解决，最老的「[问题标题]」已积攒 Y 天，要不要先处理一下？
```

## 问题提取规则

1. 识别 `数字.`、`-`、`###` 标题、`问号结尾`、`怎么做/如何/为什么/请xxx` 等模式提取独立问题
2. 同一段落如果包含多个问题（用分号、逗号分隔），拆分为独立问题
3. 去掉原文中的解释性文字、上下文铺垫，只保留问题本身
4. 去重：与 `待解决问题.md` 中已有问题做相似度比对，重复的跳过

### 内部题型判定（不在文件中显示）

提取后内部判定题型，决定后续处理方式：

| 题型 | 判定规则 | 处理方式 |
|------|---------|---------|
| 知识题 | 有标准答案的技术概念题 | 做已知/未知判定，进入问答流程 |
| 设计题 | 需要画架构图、设计方案、系统拆解 | 不做已知/未知判定 |
| 算法题 | 手写代码、算法实现 | 不做已知/未知判定 |
| 讨论题 | 开放观点题，无标准答案 | 不做已知/未知判定，标注 (讨论) |
| HR题 | 自我介绍、离职原因、职业规划等 | 归入「面试软技能」，不做已知/未知判定 |

## 二级分类体系

### 一级分类 + 二级分类完整对照表

每个问题同时标注一级分类和二级分类，格式：`[一级/二级]`，如 `[后端开发/MySQL]`。

**分类来源优先级**：源文件自带章节标题（如「## JVM 相关」）> 关键词推断。

```
AI与Agent相关
  ├── Agent架构        （ReAct、Plan-Execute、多Agent协作、状态机）
  ├── RAG              （分块、检索、重排序、评估、查询改写）
  ├── 大模型训练        （SFT、RL、CPT、微调、BERT）
  ├── 工具调用          （Function Calling、MCP、工具注册）
  ├── 记忆系统          （短期/会话/长期记忆、上下文压缩）
  ├── AI编码与SDD       （Spec Coding、OpenSpec、AI辅助开发）
  └── 多模态            （图搜、视觉、语音）

Go
  ├── 基础语法          （slice、map、defer、interface）
  ├── 并发模型          （goroutine、channel、GMP）
  └── 内存与GC

后端开发
  ├── JVM              （GC、类加载、JMM、字节码）
  ├── MySQL            （索引、事务、SQL优化、MVCC）
  ├── Redis            （数据结构、持久化、分布式锁）
  ├── 消息队列          （RocketMQ、可靠投递、重复消费）
  ├── 并发编程          （JUC、线程池、锁）
  ├── Spring           （IOC、AOP、Bean生命周期）
  └── 认证与权限        （JWT、SSO、RBAC）

系统设计
  ├── 高并发            （限流、秒杀、缓存）
  ├── 分布式            （一致性、分布式锁、CAP）
  ├── 架构设计          （微服务、系统拆分）
  └── 场景设计          （异步导出、大数据排序、长连接）

面试软技能
  ├── HR面             （离职原因、职业规划、加班看法）
  ├── 行为面            （团队冲突、技术分歧处理）
  └── 学习与成长        （学习方法、技术敏锐度）

其他技术
  ├── 操作系统与网络     （零拷贝、TCP、抓包）
  ├── Docker/K8s
  ├── 算法题
  └── 其他
```

### 分类优先级规则

1. **源文件标题优先**：如果问题来自带章节标题的面经（如「## JVM 相关」「## Agent 架构相关」），优先作为二级分类
2. **关键词权重**：Agent 相关权重最高，Go 次之，后端开发第三
3. **一个分类原则**：每个问题只归入一个二级分类
4. **兜底规则**：无法判定时标 `⚠ 待确认`，让用户手动选择分类

## 面试复盘提取

### 识别规则

遇到面经文件时，自动扫描以下特征段落：
- 标题含「面试感受」「整体感觉」「总结」等关键词
- 内容包含个人体会、公司评价、面试难度描述的段落
- 面经中非问题列表的叙述性开头/结尾

### 提取内容

1. **每轮侧重点**：从面经原文提取一面/二面/三面各自考察了什么
2. **公司风格**：面试官倾向、公司技术关注点
3. **面试者感受**：哪些模块答得顺、哪些地方被追问
4. **提示建议**：基于复盘内容，给出准备这家面经的建议

### 注意事项

- 复盘块只在「专项整理」时展示，不存储到 `待解决问题.md`
- 复盘内容不参与已知/未知判定
- 如果面经文件没有复盘内容，则跳过

---

## 已知/未知判定逻辑

### 适用范围

仅对题型标签为 `[知识]` 的问题做判定。`[设计]`、`[算法]`、`[讨论]`、`[HR]` 类问题不适用已知/未知判定。

### 判定信号来源

| 信号来源 | 判定方式 | 置信度 |
|----------|---------|--------|
| **简历-明确技能** | `简历.txt` 中列出「熟悉/熟练/深入理解」的技能 | 高 |
| **简历-项目落地** | 简历项目描述中实际用到的技术方案 | 高 |
| **已有归档** | `docs/问答归档/` 中已有同主题回答 | 中 |
| **博客笔记** | `博客学习笔记/` 中已有该主题笔记 | 中 |

### 判定分级

```
🟢 确定已知：简历明确列出 + 项目有落地
🟡 可能已知：简历提到但无项目落地，或仅归档/笔记中有记录
🔴 大概率未知：四个信号源都没有匹配
```

### 设计/讨论/算法/HR 题的特殊标注

| 题型 | 标注方式 |
|------|---------|
| [设计] | 🛠 需动手设计：无标准答案，需结合业务场景做系统设计 |
| [算法] | 💻 需编码练习：算法题，需要动手写代码 |
| [讨论] | 💬 开放讨论：无对错，需组织观点并给出论据 |
| [HR] | 👤 HR面试：非技术类，需准备个人叙事 |

### 核心原则

**宁可漏判（把已知标成待学习），不可错判（把未知标成已知）。** 所有标记为已知的都必须经用户确认后才从待解决问题中移除。

## 待解决问题.md 格式规范

```markdown
# 待解决问题

## 2026-06-03 添加

### AI与Agent/RAG
- 🟢 RAG核心流程是什么？它在电商场景中解决了哪些关键问题？
- RAG中文档分块策略有哪些？针对混合文本如何设计？
- RAG检索到语义相似但答案错误的文档如何优化？

### 后端开发/MySQL
- MySQL深度分页LIMIT 1000000,10为什么慢？有哪些优化方案？

### 面试软技能/HR面
- 你为什么离开上一家公司？

## 2026-05-20 添加

### AI与Agent/大模型训练
- SFT和RL的区别及适用场景？
```

规则：
- 按日期分组，`## YYYY-MM-DD 添加` 作为一级标题
- 日期组内按分类分组，`### 一级/二级` 作为二级标题
- 每个问题一行：`- 问题描述`
- 🟢 确定已知的问题在最前面加 `🟢`
- 讨论题末尾标注 `（讨论）`，方便区分
- 不显示题型标签、不显示来源行（信息过剩，整理报告里展示即可）
- 已解决的问题从文件中移除（用户确认后）

## 整理报告输出格式

报告由三部分组成：问题概览 + 🟢 已知标记 + 分析洞察（如果源是面经）。

```
📊 整理报告 —— {来源名称}

### 问题概览
总计 X 个问题，按分类分布：
├── AI与Agent相关：X 个（Agent架构 X、RAG X、大模型训练 X、记忆系统 X、工具调用 X、多模态 X）
├── 系统设计：X 个（高并发 X、架构设计 X、场景设计 X、分布式 X）
├── 后端开发：X 个
├── 面试软技能：X 个
└── 其他技术：X 个

### 🟢 确定已知（X 个，请手动移除）
- （AI与Agent/RAG）RAG核心流程 → 判定：简历-项目（企业知识库RAG）
- （AI与Agent/记忆系统）Agent三层记忆架构 → 判定：简历-项目（记忆压缩）
...

### 分析洞察（面经专属，无复盘内容则跳过）

#### 难度递进
一面偏知识考察，二面偏项目细节追问，三面偏技术决策和业务落地。整体难度递进明显。

#### 高频考点
- Agent架构与记忆系统：一面、二面、三面都有涉及，是这家的核心关注点
- BERT/模型优化：连续在一面和二面出现，对模型微调能力有硬性要求

#### 与用户技术栈的匹配度
- RAG、Agent架构、记忆系统 → 高匹配（简历项目直接覆盖）
- BERT微调、多模态 → 低匹配（简历无相关经验），非紧急但值得补充

#### 建议优先级
1. 先解决匹配度高的 RAG/记忆/Agent架构类（能快速产出高质量回答）
2. 再攻克匹配度低的 BERT 和多模态（补齐短板）
3. 系统设计题和讨论题穿插练习（练思维而非背答案）
```

如果源文件包含面试复盘（面试感受/整体感觉段落），在报告开头先展示复盘块：

```
### 📝 面试复盘

{原文面试感受内容}

💡 准备提示：基于复盘提炼的关键建议。
```

### 场景二（专项整理）的报告结构

当用户选「A. 技术问答归档 → 整理模式」时，整理所有待解决问题，报告额外增加：

```
⏰ 积攒超过 14 天的问题（优先解决）：
   - （2026-05-20）AI与Agent/大模型训练：大模型怎么稳定输出JSON？
```

## 收件箱处理流程

### 问题提取规则

1. 识别 `数字.`、`-`、`###` 标题、`问号结尾`、`怎么做/如何/为什么/请xxx` 等模式提取独立问题
2. 同一段落如果包含多个问题（用分号、逗号分隔），拆分为独立问题
3. 去掉原文中的解释性文字、上下文铺垫，只保留问题本身
4. 去重：与 `待解决问题.md` 中已有问题做相似度比对，重复的跳过
5. 如果收件箱内容是面经格式（有「一面/二面」章节），按面经流程处理（含题型标签 + 来源标注）

### 处理后标记

- 默认在收件箱末尾添加 `[已处理 YYYY-MM-DD]` 标记
- 不直接删除收件箱内容，保留处理记录

## 添加新问题的流程（用户直接说）

当用户直接说「加一个问题：xxx」而不走收件箱时：

1. 判断二级分类和内部题型
2. 检查是否与已有问题重复
3. 追加到 `待解决问题.md` 当天日期组对应的 `### 一级/二级` 下
4. 简短确认：已添加「问题描述」到「一级/二级」，当前总计 X 个待解决

---

## 与现有流程的集成

此 Skill 在以下时机自然介入：

- 选 A（技术问答归档）→ 检测待完成有积攒 → 问是否先整理
- 选 C（解决待完成问题）→ 按积攒天数推荐最老的问题优先
- 会话启动 → 自动检测并提醒过期问题
- 用户说「加问题」「收件箱」「整理问题」→ 直接触发对应子流程

---

## 禁止事项

1. ❌ 不要跳过收件箱、直接修改 `待解决问题.md`
2. ❌ 不要在没有用户确认的情况下删除任何问题
3. ❌ 不要将「可能已知」直接标为「确定已知」
4. ❌ 不要对设计题/算法题/讨论题/HR题做已知/未知判定
5. ❌ 不要在处理收件箱时丢失原始内容，处理后必须保留标记
6. ❌ 不要修改 `待阅读/` 中的源文件内容
7. ❌ 不要在处理面经时遗漏面试复盘内容
8. ❌ 不要在 `待解决问题.md` 中显示题型标签和来源行，文件中只保留分类+问题本身
9. ❌ 面经整理报告不要只列问题，要包含分析洞察（难度递进、高频考点、技术栈匹配度、建议优先级）
